# Wiki Schema（知识库配置规范）

> 这个文件告诉 AI 如何维护你的 Obsidian + GitHub 知识库。后续 Agent 在处理 Linux 网络、协议栈、转发面、C/C++、算法、项目和面试问题时，应优先读取本文件、`index.md`、`purpose.md` 和相关主题页。

- 创建日期：2026-06-27
- 语言：中文
- 版本：1.2

## Agent 使用规则

1. 回答和修改本知识库前，先读 `index.md`。
2. 涉及长期目标、求职方向、项目取舍时，再读 `purpose.md`。
3. 涉及具体主题时，优先读对应 Obsidian 笔记目录，再读 `wiki/` 下沉淀页。
4. 新知识不要只放聊天记录，应沉淀到最贴近的 Markdown 文件。
5. 如果是临时素材，先放 `raw/notes/` 或 `raw/articles/`，再摘要到 `wiki/sources/`。
6. 如果是稳定知识点，更新对应主题页或创建新页。
7. 如果是项目进展，更新 `08-Projects/` 和 `log.md`。
8. 如果是面试表达，更新 `09-Interview/`。
9. 如果是算法学习，更新 `11-Algorithms/` 下的学习目录或对应专题页。

## 页面命名规范

- Obsidian 主笔记：沿用现有中文文件名。
- 素材摘要：`wiki/sources/YYYY-MM-DD-短标题.md`
- 主题沉淀：`wiki/topics/主题名.md`
- 实体/概念：`wiki/entities/名称.md`
- 综合分析：`wiki/synthesis/分析主题.md`
- 对比分析：`wiki/comparisons/对比主题.md`
- 算法学习笔记：`11-Algorithms/专题名.md`

## 交叉引用规范

- 页面间使用 Obsidian 双链：`[[页面名]]`
- 涉及具体文件时，尽量链接已有 MOC 或主题页
- 每个新增 wiki 页面底部保留“相关页面”

## Ingest（消化素材）规则

处理新素材时：

1. 提取核心结论，而不是全文搬运。
2. 标注它补充了哪个能力模块。
3. 判断是否影响学习路线、项目设计或面试答案。
4. 创建或更新 `wiki/sources/` 摘要页。
5. 将稳定知识合并回主笔记。
6. 更新 `index.md` 和 `log.md`。

## Query（查询）规则

1. 先读 `index.md` 定位入口。
2. 用关键词搜索相关目录。
3. 综合已有知识库内容回答。
4. 如果发现知识库缺口，说明缺口并建议新建笔记。
5. 用户要求沉淀时，直接写回 Markdown 并提交 Git。

## Lint（健康检查）规则

定期检查：

- 是否有孤立页面。
- 是否有空笔记需要补。
- 学习路线和项目路线是否一致。
- 面试题是否能反向链接到知识点。
- `index.md` 是否覆盖新增内容。
