# `C++ 死锁` 入门：锁顺序、活锁与常见并发坑

## 先给一句话定义

死锁可以先理解成：

- **多个线程各自占着一部分资源，同时又在等对方放出另一部分资源，最后谁都走不下去**

这不是“慢”，也不是“卡一下”。
它的特点是：

- **没有进展**
- **线程互相等**
- **如果不人工干预，程序可能一直卡住**

---

## 最常见的死锁长什么样

最典型的是：

- 线程 A 拿着 `m1`，想再拿 `m2`
- 线程 B 拿着 `m2`，想再拿 `m1`

于是：

- A 等 B 放 `m2`
- B 等 A 放 `m1`

谁都不放，谁都继续不了。

---

## 一个最小例子

```cpp
#include <mutex>
#include <thread>

std::mutex m1;
std::mutex m2;

void f() {
    std::lock_guard<std::mutex> lk1(m1);
    std::lock_guard<std::mutex> lk2(m2);
}

void g() {
    std::lock_guard<std::mutex> lk2(m2);
    std::lock_guard<std::mutex> lk1(m1);
}

int main() {
    std::thread t1(f);
    std::thread t2(g);

    t1.join();
    t2.join();
}
```

这个例子的问题不是 `lock_guard`，
而是：

- **两条路径拿锁的顺序不一致**

如果时序交错成：

1. 线程 A 先拿到 `m1`
2. 线程 B 先拿到 `m2`
3. A 等 `m2`
4. B 等 `m1`

就死锁了。

---

## 死锁为什么会发生

死锁通常不是“运气差”，
而是满足了几个条件。

### 四个必要条件

1. **互斥**
   - 资源一次只能给一个线程用
2. **占有并等待**
   - 线程已经拿着一部分资源，还想再拿别的
3. **不可抢占**
   - 别的线程不能强行把资源从你手里夺走
4. **循环等待**
   - A 等 B，B 等 C，C 又等 A

这四个条件同时成立时，
死锁就有可能发生。

---

## 为什么 `lock_guard` 并不能防死锁

`lock_guard` 解决的是：

- **别忘了解锁**

它不解决的是：

- **锁和锁之间的顺序**

也就是说：

- `lock_guard` 能减少“忘记 `unlock()`”这种错误
- 但它不会自动保证“你拿两把锁时一定不会互相卡住”

所以：

- **RAII 能防泄漏，不能自动防死锁**

---

## 最重要的避免策略：统一加锁顺序

如果你的代码里需要同时拿两把锁，
最稳妥的原则就是：

- **全局统一顺序**

比如永远先锁 `m1`，再锁 `m2`。

这样即使多个线程并发，
它们也会按同一顺序排队，
就不容易形成环。

你可以把它记成：

- **锁顺序一乱，死锁概率就上来**

---

## 更安全的多锁写法：`std::scoped_lock`

如果你确实要同时锁多把 mutex，
现代 C++ 里更推荐：

```cpp
std::scoped_lock lock(m1, m2);
```

它的好处是：

- 帮你一次性拿多把锁
- 减少手写锁顺序出错

如果你还在写更老的风格，
也可以用 `std::lock` 配合 `unique_lock`，
但对当前阶段来说，
先记住：

- **多把锁优先考虑 `std::scoped_lock`**

---

## 另一个高频原则：锁内只做必要的事

很多死锁不是“真两把锁互卡”，
而是锁拿得太久，把局面变复杂了。

比如：

- 持锁做 I/O
- 持锁调用外部回调
- 持锁进入另一个会再拿锁的函数

这时就很容易把锁链条绕复杂。

所以你要尽量做到：

- **锁住共享状态的那一小段，尽快出来**

---

## 一个常见隐形坑：同一线程重复加同一把普通 mutex

普通 `std::mutex` 不是可重入锁。

也就是说：

- 同一个线程如果已经拿着它
- 又再次对它 `lock()`

通常就会把自己卡住。

这不是“多线程互相等”，
而是：

- **自己把自己堵住了**

所以在递归调用、回调重入、层层包装函数里，
这个坑很常见。

---

## 死锁和“看起来像死锁”的几个坑

### 1. 活锁

线程不是完全不动，
而是大家都在不断让步、重试、
但始终没有谁真正前进。

直觉上像：

- 你让一下
- 我让一下
- 然后都没做成事

### 2. 饥饿

某个线程长期拿不到资源，
但系统整体还在跑。

它不是“所有人都死了”，
而是：

- **有的人一直轮不到**

### 3. 忘记通知

这更像条件变量层面的卡住，
不是典型死锁，
但表现上也会像“线程一直挂着”。

### 4. 错误的锁粒度

锁太大：

- 争用严重
- 容易把外部逻辑也拖进来

锁太小：

- 保护不住完整临界区

这两边都不对。

---

## 调试时怎么看是不是死锁

如果程序表现出：

- CPU 不高
- 线程都卡住
- 某些操作永远不返回
- 日志停在某个加锁位置之后不动

那就要怀疑死锁。

排查时重点看：

1. 有没有两把以上锁
2. 锁顺序是不是在不同路径里不一致
3. 有没有持锁调用外部函数
4. 有没有同一线程重复锁同一把 mutex

---

## 最小规避清单

你现在先记这 5 条就够了：

1. 多把锁尽量固定顺序
2. 能少拿一把锁，就别多拿
3. 锁内别做慢操作
4. 别在持锁时调用不受控的外部逻辑
5. 同一线程不要重复锁同一把普通 mutex

---

## 面试够用回答

如果别人问：

“什么是死锁，怎么避免？”

你可以这样答：

死锁是多个线程互相持有对方需要的资源，形成循环等待，导致谁都无法继续执行。典型场景是多个 mutex 的加锁顺序不一致。避免方法主要有统一加锁顺序、缩小临界区、避免持锁调用外部函数，以及在需要同时拿多把锁时优先使用 `std::scoped_lock` 这类安全封装。

---

## 一句话记忆

死锁不是“代码慢了”，而是：

- **线程拿着资源不放，同时又在等别人放资源，最后形成闭环**
