# C++ 线程入门总览：执行流、同步与学习顺序

## 先给一句话定义

线程可以先理解成：

- **同一个进程里的多条执行流**

也就是说：

- 进程像一间房子
- 线程像房子里同时干活的几个人

它们共用这间房子的很多东西，
但每个人也有自己手上的那一小摊状态。

---

## 线程和进程到底差在哪

### 进程

进程通常可以先理解成：

- **资源拥有者**

它一般拥有：

- 自己的地址空间
- 打开的文件
- 各种系统资源

### 线程

线程通常可以先理解成：

- **真正执行代码的人**

同一进程里的线程通常会共享：

- 大部分内存
- 全局变量
- 堆
- 打开的文件描述符

但每个线程也有自己独立的：

- 寄存器上下文
- 调用栈

所以一句话：

- **进程更像资源边界，线程更像执行边界**

---

## 为什么要线程

线程最常见的价值有 3 类：

### 1. 并发做事

比如：

- 一个线程收网络数据
- 一个线程处理业务
- 一个线程写日志

### 2. 提高响应性

比如：

- 主线程负责界面
- 后台线程负责耗时任务

### 3. 利用多核

如果机器有多个 CPU 核，
某些任务可以真的并行跑。

---

## 线程最核心的问题，不是“怎么开”，而是“怎么共享”

新手最容易误以为线程的重点是：

- `std::thread` 怎么写

其实这只是一小层。

线程真正麻烦的地方是：

- **多个线程会碰同一份数据**

一旦共享数据，
就会立刻冒出这些问题：

1. 谁先谁后
2. 会不会同时改
3. 会不会读到一半状态
4. 会不会互相卡住

所以你后面学线程，本质是在学：

- **共享状态下的正确性**

---

## 整条学习主线，先抓这 5 个词

1. `std::thread`
2. `join` / `detach`
3. race condition（竞态条件）
4. `mutex`
5. `condition_variable`

你只要把这 5 个词串起来，线程这一章就不会散。

---

## 第一步：`std::thread` 是“把一段代码扔到新线程里跑”

最小例子：

```cpp
#include <iostream>
#include <thread>

int main() {
    std::thread t([] {
        std::cout << "worker thread\n";
    });

    t.join();
}
```

这段代码的意思是：

1. 启动一条新线程
2. 新线程执行这段 lambda
3. 主线程在 `join()` 这里等它结束

所以：

- `std::thread` 负责“开线程”
- `join()` 负责“等它跑完”

---

## 第二步：线程入口本质上就是可调用对象

你前面学的：

- 普通函数
- lambda
- `std::function`

在线程这里都会直接落地。

线程入口可以是：

- 普通函数
- lambda
- 函数对象

所以线程这一章不是新世界，
只是把前面学过的 callable 真正扔到并发场景里。

---

## 第三步：为什么会有竞态条件

看这个最小例子：

```cpp
int x = 0;

// 线程 A
x++;

// 线程 B
x++;
```

你直觉会觉得最后一定是 `2`，
但并发里不一定。

因为 `x++` 不是一个数学上的“瞬间动作”，
它底层更像：

1. 读出 `x`
2. 加一
3. 写回去

如果两个线程交错在一起，
就可能互相覆盖。

这就是：

- **race condition，竞态条件**

一句话理解：

- **多个线程同时碰同一份可变数据，结果取决于时序**

---

## 第四步：`mutex` 是“同一时刻只让一个线程进来”

这就是互斥锁的直觉。

最小例子：

```cpp
#include <iostream>
#include <mutex>
#include <thread>

int x = 0;
std::mutex m;

void work() {
    for (int i = 0; i < 100000; ++i) {
        std::lock_guard<std::mutex> lock(m);
        x++;
    }
}
```

这里你可以先粗暴理解成：

- 谁先拿到锁，谁先改
- 没拿到锁的人先等着

所以 `mutex` 解决的是：

- **共享数据同时修改**

---

## 第五步：`condition_variable` 是“光互斥还不够，我还得等条件成立”

有些问题不是简单地“别同时改”，
而是：

- 没数据时消费者要等
- 有数据了再唤醒它

这就是经典生产者消费者模型。

这里就会引出：

- `condition_variable`

它的作用不是保护共享数据本身，
而是：

- **让线程在某个条件不满足时睡下，条件满足时再被叫醒**

---

## `join` 和 `detach` 为什么总被强调

因为线程除了共享数据问题，
还有一个生命周期问题：

- 线程对象走了，线程本身怎么办

### `join`

- 当前线程等目标线程结束
- 最常见，默认先用它

### `detach`

- 线程自己在后台跑
- 原来的 `std::thread` 对象不再管理它

初学阶段先记死一句：

- **默认先 `join`，不要随手 `detach`**

---

## 线程和你前面学过的东西是怎么串起来的

### 和 lambda

线程入口最常见就是 lambda。

### 和 `std::function`

任务系统、回调系统常会用 `std::function` 装任务，
再把任务交给线程执行。

### 和智能指针

线程里经常会有对象生命周期问题，
所以：

- `unique_ptr`
- `shared_ptr`

会很常见。

### 和 `std::move`

有时候你会把对象所有权直接移动进线程入口对象里。

所以线程不是孤立章节，
而是把前面现代 C++ 那些东西真正连起来的地方。

---

## 你现在阶段，线程只要先掌握这 6 件事

1. 线程是同一进程里的多条执行流
2. `std::thread` 用来启动线程
3. `join()` 表示等待线程结束
4. 多线程共享数据会有竞态条件
5. `mutex` 用来做互斥保护
6. `condition_variable` 用来等条件成立再继续

---

## 面试够用线

如果是暑期实习，
至少先能讲顺这些：

1. 进程和线程的区别
2. `std::thread` 怎么开线程
3. `join` 和 `detach` 的区别
4. 什么是竞态条件
5. `mutex` 是干嘛的
6. `condition_variable` 典型用在什么场景
7. 死锁是怎么来的

这里暂时还不用一上来抠：

- 内存模型
- 原子操作细枝末节
- lock-free 结构

那些都更后。

## 现在这章最合适的学习顺序

我建议你就按这个顺序走：

1. 线程总览
2. `std::thread`
3. 参数传递：值 / `std::ref` / 指针
4. 竞态条件
5. `mutex` / `lock_guard`
6. `unique_lock`
7. `condition_variable`
8. 死锁和常见坑
9. 线程池是什么
10. 线程池的工程模型

线程这一章的本质不是“怎么把代码扔进新线程”，
而是：

- **多个执行流怎样安全地共享同一份状态**
