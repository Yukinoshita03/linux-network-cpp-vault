# C++ 对象的构造顺序、析构顺序与 RAII

## 先给一句话结论

在 C++ 里，最值得先记死的是：

1. **同一作用域里的局部对象，按定义顺序构造，按相反顺序析构**
2. **类成员按声明顺序构造，按相反顺序析构**
3. **基类先构造，派生类后构造；析构时相反**

这不是随便设计的，
而是为了让：

- **后构造的东西，可以安全依赖先构造的东西**

---

## 1. 同一作用域里的局部对象

看最小例子：

```cpp
#include <iostream>

struct A {
    A() { std::cout << "A ctor\n"; }
    ~A() { std::cout << "A dtor\n"; }
};

struct B {
    B() { std::cout << "B ctor\n"; }
    ~B() { std::cout << "B dtor\n"; }
};

int main() {
    A a;
    B b;
}
```

输出顺序是：

```cpp
A ctor
B ctor
B dtor
A dtor
```

也就是：

- `a` 先定义，先构造
- `b` 后定义，后构造
- 析构时反过来

这就是典型的：

- **后进先出**

---

## 2. 为什么析构顺序必须反过来

因为后构造的对象，经常依赖先构造的对象。

比如：

- `A` 先准备好某个资源
- `B` 构造时拿着 `A` 的引用、句柄、锁、连接去工作

那销毁时就必须：

1. 先销毁 `B`
2. 再销毁 `A`

不然如果先把 `A` 干掉，
`B` 的析构里还想用 `A`，就会出问题。

所以“构造正序，析构逆序”本质上是在保证：

- **依赖关系不会在清理阶段被提前切断**

---

## 3. 类成员对象也是一样

看这个：

```cpp
#include <iostream>

struct A {
    A() { std::cout << "A ctor\n"; }
    ~A() { std::cout << "A dtor\n"; }
};

struct B {
    B() { std::cout << "B ctor\n"; }
    ~B() { std::cout << "B dtor\n"; }
};

struct X {
    A a;
    B b;
};

int main() {
    X x;
}
```

顺序是：

```cpp
A ctor
B ctor
B dtor
A dtor
```

这里不是按构造函数体里的书写顺序，
而是按：

- **成员在类里声明的顺序**

---

## 4. 一个很高频的坑：不是按初始化列表顺序

很多人以为这样：

```cpp
struct X {
    A a;
    B b;

    X() : b(), a() {}
};
```

会先构造 `b`，再构造 `a`。

其实不会。

真正顺序仍然是：

1. `a`
2. `b`

因为成员构造顺序只看：

- **成员声明顺序**

不看初始化列表里谁写前谁写后。

这条很常被拿来面试问。

---

## 5. 继承场景再多一层

如果有继承：

```cpp
struct Base {
    Base() { std::cout << "Base ctor\n"; }
    ~Base() { std::cout << "Base dtor\n"; }
};

struct Member {
    Member() { std::cout << "Member ctor\n"; }
    ~Member() { std::cout << "Member dtor\n"; }
};

struct Derived : Base {
    Member m;

    Derived() { std::cout << "Derived body\n"; }
    ~Derived() { std::cout << "Derived dtor body\n"; }
};
```

构造大致顺序是：

1. `Base`
2. `m`
3. `Derived` 构造函数体

析构大致顺序是：

1. `Derived` 析构函数体
2. `m`
3. `Base`

所以可以先粗暴记成：

- **基类先活，派生类后活**
- **派生类先死，基类后死**

---

## 6. 这和 RAII 为什么直接相关

RAII 的关键就是：

- 某个对象活着时持有资源
- 某个对象死掉时释放资源

而“析构逆序”保证了：

- 后拿到的资源先释放
- 先拿到的资源后释放

这和很多真实资源模型非常匹配。

比如：

1. 先打开文件
2. 再创建缓冲区对象，缓冲区里可能还要往文件 flush

那销毁时就应该：

1. 先销毁缓冲区
2. 再关闭文件

这正是逆序析构在兜底。

---

## 7. 构造函数抛异常时，到底谁会析构

先记最核心的一句：

- **只有“已经完整构造成功”的那部分，才会被自动清理**

更精确一点拆开是：

1. 正在构造的那个**外层对象本身**，析构函数**不会**被调用
2. 这个对象里**已经构造成功的基类和成员对象**，会按逆序析构
3. 还没构造成功的那部分，不会析构
4. 如果异常继续往外抛，外层作用域里之前已经构造好的局部对象，也会在栈展开时析构

这里最容易搞混的是第 1 条：

- **构造函数一旦抛异常，说明这个对象根本没构造完成**
- **既然对象没真正“活成”一个完整对象，它自己的析构函数就不会跑**

---

## 8. 先看最简单的局部对象例子

比如：

```cpp
struct A {
    A() { std::cout << "A ctor\n"; }
    ~A() { std::cout << "A dtor\n"; }
};

struct B {
    B() {
        std::cout << "B ctor\n";
        throw std::runtime_error("fail");
    }
    ~B() { std::cout << "B dtor\n"; }
};

int main() {
    A a;
    B b;
}
```

这里会发生：

1. `A` 构造成功
2. `B` 构造时抛异常
3. `A` 自动析构
4. `B` 的析构函数**不会**执行

也就是说输出大致会像：

```cpp
A ctor
B ctor
A dtor
```

不会有：

```cpp
B dtor
```

因为 `b` 根本没构造成功。

---

## 9. 如果是“对象内部成员”构造到一半失败

看这个：

```cpp
struct M1 {
    M1() { std::cout << "M1 ctor\n"; }
    ~M1() { std::cout << "M1 dtor\n"; }
};

struct M2 {
    M2() {
        std::cout << "M2 ctor\n";
        throw std::runtime_error("fail");
    }
    ~M2() { std::cout << "M2 dtor\n"; }
};

struct X {
    M1 m1;
    M2 m2;

    X() { std::cout << "X body\n"; }
    ~X() { std::cout << "X dtor\n"; }
};
```

如果创建 `X x;`，会发生：

1. `m1` 构造成功
2. `m2` 构造时抛异常
3. `m1` 被自动析构
4. `m2` 不析构
5. `X` 本身的析构函数也不执行

所以这时你要分清三层：

- 已成功的成员：会清理
- 失败中的成员：不会清理
- 外层对象自己：也不会清理

---

## 10. 如果异常是在构造函数体里抛的

再看另一种情况：

```cpp
struct X {
    M1 m1;
    M2 m2;

    X() {
        std::cout << "X body\n";
        throw std::runtime_error("fail");
    }

    ~X() { std::cout << "X dtor\n"; }
};
```

这次情况是：

1. `m1` 已经构造成功
2. `m2` 也已经构造成功
3. 进入 `X` 的构造函数体
4. 构造函数体里抛异常
5. `m2`、`m1` 按逆序析构
6. `X` 自己的析构函数仍然**不会**执行

所以判断标准不是“有没有进到构造函数体”，
而是：

- **这个完整对象是否构造完成**

只要 `X` 的构造函数没有成功走完，
`X` 就不算完整构造成功，
于是 `~X()` 不会跑。

---

## 11. 这和 `new` 还有一层容易混

比如：

```cpp
X* p = new X();
```

如果 `X()` 构造时抛异常：

- `X` 对象本身没有构造成功
- `X` 的析构函数不会跑
- 但这次 `new` 申请到的那块原始内存，运行时会负责回收

所以单纯写：

```cpp
new X()
```

如果构造函数抛异常，**对象内存本身通常不会因为这条 `new` 表达式而泄漏**。

但要注意一个更阴的点：

- **如果你在构造函数里手动拿了别的资源，却没交给成员对象管理，那还是可能泄漏**

比如：

```cpp
struct Bad {
    int* p;

    Bad() {
        p = new int[100];
        throw std::runtime_error("fail");
    }
};
```

这里 `Bad` 自己没构造成功，
所以 `~Bad()` 不会跑。
如果 `p` 只是你手动记在裸指针里，
那这块 `new int[100]` 就没人收了。

这正是 RAII 真正值钱的地方：

- **把资源放进成员 RAII 对象里**
- **这样即使构造中途抛异常，已经成功构造的成员也会自动清理**

---

## 12. 这就是为什么 RAII 对异常安全特别强

- **只要资源被装进对象里，构造成功过的那部分就能自动清理**

反过来说：

- 如果资源只是你在构造函数体里手搓、手记、手动管
- 而不是交给已经构造好的成员对象

那构造失败时就容易漏清理。

---

## 13. 你现在该怎么记

先记这 5 句就够了：

1. 局部对象按定义顺序构造，按逆序析构。
2. 成员对象按声明顺序构造，按逆序析构。
3. 初始化列表顺序不决定成员构造顺序。
4. 基类先构造，派生类后构造；析构相反。
5. 逆序析构的目的，是保证依赖关系在清理阶段仍然成立。

如果再补 4 句异常版，就是：

6. 构造函数抛异常时，外层对象自己不算构造成功，所以它自己的析构函数不会跑。
7. 已经构造成功的成员和基类会按逆序析构。
8. 没构造成功的那部分不会析构。
9. RAII 的关键是把资源放进成员对象里，而不是等构造函数最后手动收尾。

---

## 一句话收住

“构造正序，析构逆序”不是死记语法，
而是 C++ 为了让对象依赖、异常展开和资源清理天然成立的底层规则。
