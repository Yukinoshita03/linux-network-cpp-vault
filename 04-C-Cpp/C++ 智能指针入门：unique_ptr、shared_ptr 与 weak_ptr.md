# C++ 智能指针入门：unique_ptr、shared_ptr 与 weak_ptr

## 先给一句话定义

智能指针可以先理解成：

- **会自动管理资源释放的对象**

更准确一点说：

- 它不是“更聪明的裸指针”
- 而是“把所有权规则写进类型里”的 RAII 工具

所以你学智能指针，真正学的不是指针语法，
而是：

- **对象到底归谁管，什么时候该释放**

---

## 为什么 C++ 要搞智能指针

因为手写 `new` / `delete` 很容易出事：

1. 忘记 `delete`
2. 提前 `delete`
3. 多次 `delete`
4. 异常路径没清理
5. 谁负责释放说不清

智能指针解决的核心问题是：

- **把“释放责任”绑定到对象生命周期上**

这就是 RAII 那套思路。

所以准确说，不是“智能指针负责帮你申请内存”，
而是：

- **动态分配出来的对象，立刻交给智能指针接管**

---

## 先记结论：三兄弟分别管什么

### `unique_ptr`

- 独占所有权
- 一个资源同一时刻只归一个人管

### `shared_ptr`

- 共享所有权
- 多个人一起管，靠引用计数决定最后何时释放

### `weak_ptr`

- 弱引用，不拥有对象
- 用来观察 `shared_ptr` 管的对象，避免循环引用

---

## 1. `unique_ptr`：默认应该先想到它

```cpp
auto p = std::make_unique<int>(42);
```

### 它的核心语义

`unique_ptr` 的核心不是“自动释放”四个字，
而是：

- **独占**

也就是：

- 这块资源只有一个明确 owner

所以它：

- 不能拷贝
- 但能移动

```cpp
auto p1 = std::make_unique<int>(42);
auto p2 = std::move(p1);
```

这和你前面学的移动语义是完全连起来的。

移动之后：

- 所有权从 `p1` 转给 `p2`
- `p1` 变成空的

### 为什么它很重要

因为大多数资源其实天然就是：

- **单一所有者**

这时用 `unique_ptr` 最清楚、最便宜。

---

## 2. `shared_ptr`：多个 owner 共同持有

```cpp
auto p1 = std::make_shared<int>(42);
auto p2 = p1;
```

这里发生的是：

- `p1` 和 `p2` 一起拥有同一个对象

背后通常靠：

- **引用计数**

来管理生命周期。

你可以粗略理解成：

- 每多一个 `shared_ptr` 拷贝，计数加一
- 每少一个，计数减一
- 计数归零时对象释放

### 它的好处

适合这种场景：

- 对象真的需要被多个地方共同持有
- 你很难明确说谁一定活得更久

### 它的代价

它不是白送的。

通常要付出：

- 引用计数维护成本
- 所有权关系更复杂
- 更容易写出循环引用

所以不要把 `shared_ptr` 当默认选项。

---

## 3. `weak_ptr`：只看，不拥有

`weak_ptr` 是配合 `shared_ptr` 出现的。

它表示：

- 我想观察这个对象
- 但我不参与拥有它

所以：

- `weak_ptr` 不增加引用计数

### 为什么它存在

因为 `shared_ptr` 最经典的坑就是：

- **循环引用**

比如 A 持有 B 的 `shared_ptr`，
B 也持有 A 的 `shared_ptr`。

那结果就是：

- A 等 B 先死
- B 等 A 先死
- 谁都死不了

这时就需要有一边改成：

- `weak_ptr`

打破这个环。

### 怎么用

`weak_ptr` 不能直接像 `shared_ptr` 那样随便解引用，
通常要先：

```cpp
if (auto sp = wp.lock()) {
    // 说明对象还活着
}
```

`lock()` 成功就拿到一个临时的 `shared_ptr`，
失败说明对象已经没了。

---

## 智能指针真正要学的是“所有权模型”

这比 API 本身重要得多。

### 情况 1：根本不需要堆对象

那就别上智能指针，
直接普通对象：

```cpp
Widget w;
```

### 情况 2：需要堆对象，但只有一个 owner

优先：

- `unique_ptr`

### 情况 3：确实有多个 owner 要共享生命周期

才考虑：

- `shared_ptr`

### 情况 4：只想引用，不想拥有

考虑：

- `weak_ptr`
- 或普通裸指针 / 引用作为观察者

---

## 为什么总说“默认先想 unique_ptr”

因为它：

1. 语义最清楚
2. 性能负担最小
3. 不容易把所有权搞乱
4. 和移动语义天然契合

所以一句话：

- **能不共享，就别共享**

---

## 你前面学的 `move`，在这里怎么落地

看这个：

```cpp
std::unique_ptr<Node> p = std::make_unique<Node>();
vec.push_back(std::move(p));
```

这里 `move` 的意思非常真实：

- 把所有权交给容器

这时你前面学的“`std::move` 只是转换，不保证性能提升”那套，
在智能指针场景里会变得非常直观：

- `unique_ptr` 之所以能转移，是因为它的移动语义本来就代表所有权转移

---

## 新手最容易踩的 5 个坑

### 1. 明明单一 owner，却上来就用 `shared_ptr`

这会把本来简单的问题搞复杂。

### 2. 从同一个裸指针构造多个 `shared_ptr`

这是危险写法。

因为你等于让多个独立控制块都以为自己该负责释放同一个对象。

### 3. 还在手写 `new`

优先：

```cpp
std::make_unique<T>(...)
std::make_shared<T>(...)
```

而不是：

```cpp
std::unique_ptr<T>(new T(...))
```

### 4. `weak_ptr` 当成普通指针直接用

不行。

它要先 `lock()`。

### 5. 以为智能指针能代替一切生命周期设计

不能。

它只是把规则写清，
不是替你自动想清楚“谁该拥有谁”。

---

## 一张选择表

| 场景 | 更自然的选择 |
|---|---|
| 不需要动态分配 | 普通对象 |
| 动态分配，单一 owner | `unique_ptr` |
| 动态分配，多个 owner | `shared_ptr` |
| 只观察，不拥有 | `weak_ptr` / 引用 / 裸观察指针 |

---

## 这一页真正该记住的 6 句话

1. 智能指针本质上是在表达所有权
2. `unique_ptr` 表示独占所有权，不能拷贝，只能移动
3. `shared_ptr` 表示共享所有权，背后通常有引用计数
4. `weak_ptr` 不拥有对象，只是观察者
5. `shared_ptr` 最大的经典问题是循环引用
6. 默认先想 `unique_ptr`，不要把 `shared_ptr` 当默认

---

## 面试够用线

至少要能顺着讲清：

1. 智能指针解决的是资源释放和所有权表达问题
2. `unique_ptr` 是独占所有权，适合默认场景
3. `shared_ptr` 是共享所有权，但有引用计数成本和循环引用风险
4. `weak_ptr` 用来观察 `shared_ptr` 管的对象，并打破环
5. 优先用 `make_unique` / `make_shared`

---

## 下一步最自然接什么

学完智能指针之后，
如果继续沿着现代 C++ 常用能力往前走，
下一步最自然的是：

1. `std::function`
2. 线程与并发基础

因为后面你会经常看到：

- lambda + 回调
- 智能指针 + 异步生命周期

这两个组合。
