# `vector` 的 `insert` 与 `erase` 为什么贵

## 为什么这页重要

很多人背过一句话：

- `vector` 适合尾插
- 不适合中间插入删除

但如果不知道“为什么”，就只能死记。  
本质原因只有一个：

- `vector` 必须维护**连续内存**

## 先抓最核心的画面

假设现在有：

```text
[ A ][ B ][ C ][ D ][ E ]
```

如果你要在 `C` 前面插入一个 `X`：

```text
[ A ][ B ][ X ][ C ][ D ][ E ]
```

那 `C D E` 原来占的位置不够了，所以必须整体向后挪一格。

这就是 `insert` 贵的根因。

如果你删掉 `C`：

```text
[ A ][ B ][ D ][ E ]
```

为了继续保持连续，`D E` 又必须整体向前补洞。

这就是 `erase` 贵的根因。

## 1. `insert` 到底做了什么

看：

```cpp
v.insert(v.begin() + k, value);
```

底层大致要做两件事。

### 情况 A：容量不够

这时要先扩容：

1. 分配更大内存
2. 把旧元素搬到新内存
3. 在新位置插入新元素
4. 销毁旧元素并释放旧内存

这已经很贵了。

### 情况 B：容量够

就算不扩容，也还要：

1. 把 `[k, size)` 这一段元素整体后移
2. 在第 `k` 个位置构造新元素

所以复杂度仍然是：

- O(n)

不是因为重新分配，而是因为要移动后半段元素。

## 2. `erase` 到底做了什么

看：

```cpp
v.erase(v.begin() + k);
```

这时一般不会扩容，但会：

1. 销毁第 `k` 个元素
2. 把 `[k+1, size)` 这一段整体前移一格

所以复杂度也还是：

- O(n)

因为后半段要整体搬家。

## 3. 为什么尾部操作便宜

### `push_back`

如果没扩容：

- 直接在尾部构造元素

不需要动前面的元素。

### `pop_back`

- 直接销毁最后一个元素

也不需要动前面的元素。

所以尾部操作通常便宜。

## 4. 真正贵在哪里

不要只记“O(n)”，要知道成本主要由什么组成。

### 成本 1：元素移动 / 拷贝

如果元素类型很轻，比如 `int`，移动代价还好。

如果元素类型很重，比如：

- `std::string`
- 自定义大对象
- 资源管理类

那么后半段每个元素的移动/拷贝成本都可能不低。

### 成本 2：可能触发扩容

如果插入时容量还不够，还要先走整轮扩容搬迁。

### 成本 3：缓存和内存写入

连续移动一大段元素，会产生实际的内存读写成本。  
虽然 `vector` cache 友好，但“大段搬运”本身仍然是成本。

## 5. 为什么 `erase` 返回迭代器

因为删掉一个位置以后，后面的元素整体前移，旧迭代器失效。

所以库直接把：

- 删除后当前位置的下一个有效迭代器

返回给你。

典型写法：

```cpp
it = v.erase(it);
```

这样可以继续安全遍历。

## 6. 为什么边遍历边删容易写错

因为很多人会这样：

```cpp
for (auto it = v.begin(); it != v.end(); ++it) {
    if (*it == 0) {
        v.erase(it);  // 错误高发
    }
}
```

问题在于：

- `erase(it)` 之后，`it` 已经失效
- 循环结尾再 `++it`，就可能出问题

更稳的写法：

```cpp
for (auto it = v.begin(); it != v.end(); ) {
    if (*it == 0) {
        it = v.erase(it);
    } else {
        ++it;
    }
}
```

## 7. 和别的容器相比，为什么这是 `vector` 的宿命

因为 `vector` 的最大优势就是：

- 连续内存
- 随机访问快
- cache 友好

而中间插入删除贵，正是这个设计的代价。

也就是说：

- 不是 STL 写得不好
- 而是数据结构本身就在做这个 trade-off

## 8. 一句话总结

`vector` 的 `insert` / `erase` 贵，不是因为接口复杂，而是因为它必须维护连续内存；中间动一个位置，就要整体搬后面那一段元素。
