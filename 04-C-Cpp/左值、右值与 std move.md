# 左值、右值与 `std::move`

## 最容易犯的错

左值和右值，**不是**简单指：

- 在等号左边的值
- 在等号右边的值

那只是非常早期、非常粗糙的记忆法，深入到移动语义时基本不够用。

## 更准确的理解

### 左值

左值可以先理解成：

- 有名字
- 有稳定身份
- 之后还可能继续被使用

比如：

```cpp
int x = 10;
std::string s = "hello";
```

这里的 `x` 和 `s` 都是左值。  
因为它们代表的是“一个还活着、之后还会继续用的对象”。

### 右值

右值可以先理解成：

- 临时的
- 即将消亡的
- 没有必要长期保留原状态

比如：

```cpp
std::string("hello")
```

这是一个临时对象。它通常就是右值。

## 为什么移动语义喜欢右值

因为右值通常表示：

- 这个对象反正快没用了
- 那我可以放心把它内部资源拿走

所以移动语义的核心直觉不是：

- “它在等号右边”

而是：

- “它像一个即将被丢弃的对象”

## 你的说法哪里接近了

你说：

> 他这个状态像右值一样

这一半是对的。

因为 `std::move(x)` 的作用，确实是把 `x` 变成一种：

- “请把我当成快要被搬空的对象来看”

但你后面说：

> 是要被给左边的

这个说法就不准确了。

右值不是因为“要赋值给左边”，而是因为：

- 它表示一个临时值或将亡值
- 编译器和标准库可以据此选择移动构造/移动赋值

## `std::move` 到底做了什么

`std::move(x)` 不是移动本身，它只是：

- 把原本是左值的 `x`
- 强制转换成一个“将亡值”表达式

也就是：

- 它告诉后续代码：`x` 现在可以被当成可移动对象处理

例如：

```cpp
std::string s1 = "hello";
std::string s2 = std::move(s1);
```

这里不是 `std::move` 把资源搬走了，而是：

1. `std::move(s1)` 把 `s1` 变成一个可移动的右值表达式
2. `string` 的移动构造函数看到它是右值
3. 于是把 `s1` 的资源转移给 `s2`

## 为什么 `std::move(x)` 之后也可能最后还是拷贝

因为 `std::move` 只负责：

- 把 `x` 转成一个可移动的右值表达式

它**不负责保证目标类型一定存在可用的移动构造/移动赋值**。

后面到底走移动还是拷贝，取决于：

- 这个类型有没有移动构造
- 这个移动构造能不能被调用
- 当前上下文接收的是构造还是赋值

### 典型情况 1：类型根本没有可用的移动构造

如果一个类型没有移动构造，或者移动构造被禁用，那么即使你写了 `std::move(x)`，后续也可能退回去走拷贝构造。

### 典型情况 2：对象是 `const`

这是非常高频、也最容易忽略的一种。

```cpp
const std::string s = "hello";
std::string t = std::move(s);
```

这里很多人以为会移动，其实通常不会得到真正的移动语义。  
原因是：

- 移动往往需要修改源对象
- 但 `const` 对象不允许被修改

所以它通常只能拷贝。

你可以把它理解成：

- `std::move(const T&)` 只是把它变成“`const` 右值”
- 但“`const` 右值”通常没法交给需要修改源对象的移动构造去处理

### 典型情况 3：你以为自己写了移动构造，其实它并不是真正便宜的移动

比如你写了一个“移动构造”，但内部还是：

- 分配新内存
- 复制全部数据

那语法上虽然走的是移动构造，性能上却并没有真正获得“资源转移”的好处。

所以“走移动构造”和“真的便宜”不是一回事。

## 最该记住的一句话

`std::move` 只是给了“可以尝试移动”的资格；真正能不能移动、会不会退回拷贝、移动是否便宜，取决于类型本身。

## 为什么普通变量默认不会自动移动

因为普通变量默认是左值。

编译器默认认为：

- 你后面可能还要继续用它

所以它不会擅自把一个左值当成“可被搬空”的对象。  
这就是为什么要显式写：

```cpp
std::move(x)
```

本质上你是在说：

- 我知道 `x` 之后不再需要原状态了
- 你可以考虑搬走它的资源

## 一个常见但不够准确的说法

很多人会说：

- “`std::move` 让它变成即将死亡的值”
- “`std::move` 让它变成要释放的对象”

这两个说法都不够准确。

更准确的表达是：

- `std::move` 让一个左值变成“可以按将亡值对待”的表达式
- 它表示“这个对象的资源现在可以被转移”
- 它**不等于**这个对象马上销毁
- 它**不等于**立刻释放资源

例如：

```cpp
std::string s1 = "hello";
std::string s2 = std::move(s1);
```

这里发生后：

- `s1` 还活着
- `s1` 仍然是合法对象
- 只是它的内容状态通常不再是原来的 `"hello"`

所以 `std::move` 更像是在说：

- “我允许你把 `s1` 当成资源可被转移的对象”

而不是：

- “请马上销毁 `s1`”
- “请马上释放 `s1`”

## 一句话记忆

左值像“还要继续用的对象”，右值像“临时值或将亡值”；`std::move` 的作用是把左值伪装成“可以被搬走的将亡值”。

## 更正式一点的版本：左值、纯右值、将亡值

如果往正式术语靠一点，C++ 里常见的值类别可以先这样理解：

### 1. 左值（lvalue）

特点：

- 有稳定身份
- 可以取地址
- 通常代表一个还会继续存在的对象

例子：

```cpp
int x = 1;
x;          // 左值
std::string s = "hi";
s;          // 左值
v[0];       // 通常也是左值
```

### 2. 纯右值（prvalue）

特点：

- 更像一个纯粹的临时值
- 常常是字面量、临时对象、表达式结果

例子：

```cpp
42;                     // 纯右值
std::string("hello");   // 纯右值
x + 1;                  // 纯右值
```

### 3. 将亡值（xvalue）

这是和移动语义最相关的那类值。

特点：

- 仍然和某个对象有关
- 但这个对象的资源可以被安全转移

例子：

```cpp
std::move(s)
```

这里 `std::move(s)` 最值得你记住的正式理解是：

- 它产生的是一种“将亡值”表达式

也就是：

- 这个对象还存在
- 但它现在可以被当成资源可转移对象来处理

## 为什么不用死记全部术语

你当前阶段其实不用死背：

- glvalue
- prvalue
- xvalue

但至少要知道这件事：

`std::move(x)` 不是把 `x` 变成普通“临时值”，而是更准确地把它变成一种：

- **将亡值**

这就是为什么它和移动构造能接上。

## 你现在最够用的三层模型

### 日常直觉层

- 左值：后面还要用
- 右值：临时的，可搬

### 机制层

- 左值默认不会触发移动
- `std::move` 把左值转成可移动表达式

### 正式层

- `std::move(x)` 常见地生成将亡值（xvalue）
- 移动构造 / 移动赋值就是专门接这种值类别的

## 为什么这和 `vector` 有关

因为：

```cpp
v.push_back(x);            // x 是左值
v.push_back(std::move(x)); // std::move(x) 是将亡值表达式
```

前者更容易走拷贝路径，后者才给了移动构造参与的机会。
