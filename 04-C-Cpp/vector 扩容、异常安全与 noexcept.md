# `vector` 扩容、异常安全与 `noexcept`

## 为什么这页重要

前面已经知道：

- `vector` 扩容会整体搬家
- 搬家时可能调用移动构造或拷贝构造
- `noexcept` 会影响标准库更敢不敢走移动路径

这一页把三件事串起来：

1. 扩容到底分几步
2. 为什么标准库很在意异常安全
3. 为什么 `noexcept` 会改变真实行为

## 1. `vector` 扩容的真实流程

极简模型：

```cpp
if (size == capacity) {
    申请更大内存;
    把旧元素搬到新内存;
    销毁旧内存里的旧元素;
    释放旧内存;
}
在尾部构造新元素;
```

重点不在“申请更大内存”，而在：

- **搬旧元素这一步可能失败**

## 2. 为什么搬旧元素可能失败

因为元素类型 `T` 的构造过程不一定安全。

比如：

- 拷贝构造可能分配内存
- 移动构造也可能写得很复杂
- 自定义类型构造里可能抛异常

所以标准库要面对的问题是：

- 如果搬到一半抛异常，容器还能不能保持正确状态？

## 3. 异常安全在这里到底想保什么

最理想的目标是：

- 如果扩容失败，原来的 `vector` 还是原样

这就是你可以先理解成的“强保证”。

也就是说：

- 要么整个扩容成功
- 要么像什么都没发生过

这对容器用户非常友好。

## 4. 为什么拷贝有时反而更稳

如果用拷贝把旧元素迁移到新内存：

1. 旧内存里的原对象还没被破坏
2. 新内存里只是逐步生成副本
3. 如果中途失败，把新内存那部分清掉即可
4. 旧 `vector` 还保持原样

所以从异常安全角度看：

- 拷贝路径通常更容易保住“原容器不变”

## 5. 为什么移动有时反而让库更保守

如果用移动搬元素：

1. 一部分旧元素已经搬走
2. 那些旧元素可能已经进入“被移动后状态”
3. 中途再抛异常，旧容器就不再是原样

这时回滚会更麻烦。

所以标准库不是“讨厌 move”，而是：

- **怕可能抛异常的 move**

## 6. `noexcept` 在这里起什么作用

如果移动构造是：

```cpp
T(T&&) noexcept
```

那标准库就更敢判断：

- 这次搬旧元素时，移动这一步不会抛异常

于是它更愿意选移动路径。

所以：

- `noexcept` 不是为了语法好看
- 它是在降低容器搬迁时的风险判断成本

## 7. 一个接近真实的选择逻辑

你可以先把库里的策略理解成：

```cpp
if (移动构造是 noexcept || 类型不能拷贝) {
    用移动构造搬旧元素;
} else {
    用拷贝构造搬旧元素;
}
```

这不是标准条文原样，但很接近你现在该掌握的行为模型。

## 8. 为什么“不能拷贝时只能 move”

如果某个类型：

- 不可拷贝
- 只能移动

那标准库即使知道移动可能抛异常，也没太多选择空间。  
这时往往只能走移动路径。

所以你要把逻辑记成：

1. 能安全 move，就 move
2. move 风险高而 copy 可用，就优先 copy
3. copy 根本不可用，那就只能 move

## 9. 这和性能怎么联系起来

如果一个类型：

- 深拷贝很贵
- 移动很便宜
- 且移动构造还是 `noexcept`

那它对 `vector` 非常友好。  
因为扩容时，标准库更可能：

- 走高效移动路径
- 同时不破坏异常安全目标

反过来，如果一个类型：

- 拷贝贵
- 移动也可能抛异常

那 `vector` 在扩容时就更尴尬：

- 要么保守 copy，很慢
- 要么 move 但异常安全更难处理

## 10. 一句话总结

`vector` 扩容真正难的不是“申请大内存”，而是“搬旧元素时如何既快又稳”；`noexcept` 之所以重要，就是因为它让标准库更敢选择高效移动而不牺牲异常安全。

## 11. 对应这条分支，分别长什么样

你前面看到的理解模型是：

```cpp
if (移动构造是 noexcept || 根本不能拷贝) {
    用移动构造搬旧元素;
} else {
    用拷贝构造搬旧元素;
}
```

下面用 3 类最典型的类型来看。

### 情况 A：能拷贝，而且移动构造是 `noexcept`

这是标准库最舒服的一类。

```cpp
class Buffer {
public:
    Buffer(size_t n)
        : size_(n), data_(new int[n]) {}

    ~Buffer() { delete[] data_; }

    Buffer(const Buffer& other)
        : size_(other.size_), data_(new int[other.size_]) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    Buffer(Buffer&& other) noexcept
        : size_(other.size_), data_(other.data_) {
        other.size_ = 0;
        other.data_ = nullptr;
    }

private:
    size_t size_;
    int* data_;
};
```

如果有：

```cpp
std::vector<Buffer> v;
```

扩容时，库更可能选：

- move

因为：

- move 很便宜
- 而且 `noexcept`
- 没必要保守地走 copy

### 情况 B：能拷贝，但移动构造可能抛异常

这时库就更保守。

```cpp
class Risky {
public:
    Risky() = default;

    Risky(const Risky&) {
        // 假设这里是可用的拷贝路径
    }

    Risky(Risky&&) {
        // 假设这里做复杂逻辑，可能抛异常
        may_throw();
    }

private:
    void may_throw();
};
```

如果有：

```cpp
std::vector<Risky> v;
```

扩容时，库往往更倾向：

- copy

原因是：

- move 可能把一半旧对象搬空后再抛异常
- copy 更容易保住“失败时原容器不变”

这就是“明明有移动构造，为什么还可能选拷贝”的典型例子。

### 情况 C：根本不能拷贝，只能移动

这时库没得选。

最经典的是：

```cpp
std::vector<std::unique_ptr<int>> v;
```

因为 `unique_ptr`：

- 禁止拷贝
- 只能移动

所以扩容时：

- 不管你喜不喜欢
- 只能 move

这就是前面分支里：

```cpp
|| 根本不能拷贝
```

这半句存在的现实意义。

## 12. 用这 3 类再记一遍分支

### A 类：能 copy，move 还 `noexcept`

- 选 move

### B 类：能 copy，但 move 可能抛异常

- 常常选 copy

### C 类：不能 copy

- 只能 move

## 13. `int` 和 `std::string` 的搬迁差别

先纠正一句表述：

- `int` 是类型
- `std::string` 也是类型

真正的区别不是“一个是类型，一个不是类型”，而是：

- `int` 这类通常是平凡类型，拷贝、搬迁、销毁都很简单
- `std::string` 是非平凡类类型，有构造、析构、移动/拷贝语义，内部通常还管理字符缓冲区等状态

所以你可以粗暴地先这样理解：

- `vector<int>` 扩容时，底层实现常常可以把搬迁优化得非常接近“按字节搬”
- `vector<std::string>` 扩容时，语义上必须按“对象”来搬，而不是把内存当成一坨裸字节乱拷

对 `vector<std::string>` 来说，扩容更接近下面这个过程：

1. 申请一块更大的新内存
2. 在新内存上对每个旧元素调用移动构造或拷贝构造
3. 把旧内存位置上的原 `string` 对象析构掉
4. 释放旧 buffer

为什么不能把 `std::string` 直接 `memcpy` / `memmove` 完事？

- 因为它不是单纯的值位模式
- 它内部往往还有指针、长度、容量，甚至 small string optimization 这类实现细节
- 如果只是粗暴复制字节，两个对象可能会错误地共享同一份内部资源
- 最后就可能出现双重释放、悬空指针或破坏对象不变量

所以更准确的一句话应该是：

- `int` 这类平凡类型，常可以近似理解成“搬字节”
- `std::string` 这类非平凡对象类型，必须按对象语义搬

这也是为什么元素类型越“重”，`vector<T>` 扩容成本越明显。
