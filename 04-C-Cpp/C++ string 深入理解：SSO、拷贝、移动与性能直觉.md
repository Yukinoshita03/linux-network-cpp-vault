# `std::string` 深入理解：`SSO`、拷贝、移动与性能直觉

## 为什么这页重要

学到移动语义后，很多人会对 `std::string` 产生一个很自然的印象：

- 拷贝就是复制字符
- 移动就是偷指针

这个印象只对了一半。

一旦把 `SSO` 也考虑进来，你就会发现：

- `string` 的移动通常比拷贝便宜
- 但对短字符串，它未必像你想象中那么“神”

这页就是把这件事讲透。

## 先别把 `string` 当成“就是一坨字符”

`std::string` 不是 `char*`，也不是单纯的字符数组。

它本质上是一个**类对象**，负责管理一段字符串数据。

你可以先把它粗略理解成这样：

```cpp
class string_like {
    char* ptr;
    size_t size;
    size_t capacity;
};
```

这只是帮助理解的模型，不是标准规定的真实布局。

真实实现往往更复杂，因为标准库还会做优化，其中最有名的就是：

- `SSO`

## 什么是 `SSO`

`SSO` 是 `Small String Optimization`，小字符串优化。

核心思想是：

- 如果字符串很短
- 那就不要单独去堆上申请内存
- 直接把字符放在 `string` 对象内部

也就是说，`std::string` 常常有两种工作模式。

### 模式 A：长字符串，堆分配

比如字符串比较长时：

- 对象里放的是指针、长度、容量
- 真正的字符数据在堆上

可以粗略想成：

```text
string 对象
+----------------------+
| ptr -----> heap data |
| size                 |
| capacity             |
+----------------------+
```

### 模式 B：短字符串，`SSO`

如果字符串很短：

- 不分配堆内存
- 字符直接存在对象内部的小缓冲区里

可以粗略想成：

```text
string 对象
+------------------------------+
| inline buffer: "abc"         |
| size                         |
| 其他状态                     |
+------------------------------+
```

所以关键点是：

- 长字符串的数据在对象外面
- 短字符串的数据就在对象里面

## `SSO` 为什么重要

因为它直接改变了你对 `copy` 和 `move` 的性能直觉。

如果没有 `SSO`，你很容易形成一个简单模型：

- copy：重新申请内存，再复制字符
- move：把原来的指针拿过来

对很多长字符串，这个模型很接近真实情况。  
但对短字符串，就没这么简单了。

## 先看长字符串：为什么 move 通常很便宜

假设：

```cpp
std::string s = "this is a very long string ...";
```

如果它已经走到堆分配模式，那么：

### 拷贝

```cpp
std::string t = s;
```

更接近：

1. 给 `t` 申请一块新的字符内存
2. 把 `s` 的字符内容复制过去
3. `s` 和 `t` 各自持有自己的数据

### 移动

```cpp
std::string t = std::move(s);
```

更接近：

1. `t` 接管 `s` 原来指向的那块堆内存
2. `s` 被重置成一个有效但值未指定的状态

所以对长字符串来说：

- copy 通常要真的复制一份字符数据
- move 往往只是在转交资源控制权

这就是“移动像偷指针”这个直觉成立的地方。

## 再看短字符串：为什么 move 未必神奇

假设：

```cpp
std::string s = "abc";
```

如果 `"abc"` 走的是 `SSO` 模式，那么字符根本不在堆上，而是在 `s` 这个对象内部。

这时如果你写：

```cpp
std::string t = std::move(s);
```

问题来了：

- `s` 里的字符是存在 `s` 自己对象内部的
- `t` 是另一个对象
- `t` 不可能“偷走 `s` 对象内部那块物理内存”

因为那块内存就是 `s` 自己身体的一部分，不是独立堆资源。

所以对短字符串，移动常常更接近：

1. 把那几个内嵌字符复制到 `t` 自己的内嵌缓冲区里
2. 再把 `s` 调整成已移动状态

这时你就会看到一个很重要的现象：

- 语义上它还是 move
- 但性能上它未必比 copy 快很多

因为两边都只是处理很少几个字节。

## 这就是 `SSO` 对直觉的冲击

你现在应该把 `std::string` 分成两类看。

### 长字符串

- 常常涉及堆分配
- copy 往往更贵
- move 往往很值钱

### 短字符串

- 常常直接放对象内部
- copy 本来就不贵
- move 也可能只是搬几字节小数据

所以：

- `move` 仍然有语义价值
- 但不一定总有你想象中的巨大性能优势

## 为什么标准库要做 `SSO`

因为现实代码里短字符串非常多，比如：

- `"ok"`
- `"GET"`
- `"name"`
- `"json"`
- 很多配置项、字段名、状态名

如果这些小字符串每次都去堆上分配：

- 申请内存会有开销
- 释放内存会有开销
- 内存碎片和缓存局部性也更差

所以 `SSO` 的核心收益是：

- 少一次堆分配
- 少一次堆释放
- 小字符串操作更轻

## 这和 `vector<string>` 有什么关系

这就是前面那句“`vector<string>` 扩容不总是你想象中那么重”的原因。

### 如果元素多是长字符串

那么扩容时：

- copy 可能很贵
- move 通常很有价值

### 如果元素多是短字符串

那么扩容时即便走 move：

- 也可能只是把对象内部那一点小数据搬过去
- 优势可能没有“偷大块堆内存”那么夸张

但仍然要记住：

- `vector<string>` 扩容搬的依然是对象
- 不是把整片内存当成裸字节直接乱搬

## 高频纠偏

### 1. `std::string` 的移动不等于永远零成本

不对。

更准确地说：

- 它通常比深拷贝便宜
- 但对短字符串，可能只是“便宜一点”，不是“神一样地便宜”

### 2. moved-from 的 `string` 一定是空串吗

不一定。

标准只保证：

- 它仍然是有效对象
- 但值未指定

很多实现里你会看到它像空串，但不要把这当成必须行为。

### 3. `SSO` 的阈值是不是固定的

不是。

这取决于具体标准库实现和平台。

所以你应该记：

- 有 `SSO` 这件事很常见
- 但“最多多少字符走 `SSO`”不是标准保证

### 4. `string` 的真实内存布局能不能写死去依赖

不能。

你可以把对象模型当成理解工具，但不要把某个实现的内部布局当成可移植契约。

## 面试里怎么说最稳

如果别人问你：

“为什么 `std::string` 的 move 不一定像你想象中那么快？”

你可以这样答：

1. `std::string` 往往有小字符串优化 `SSO`
2. 长字符串的数据在堆上，move 常常能直接转交堆资源，所以很便宜
3. 短字符串的数据在对象内部，目标对象没法直接偷走源对象内部那块内存
4. 所以短字符串的 move 可能也要复制那几个内嵌字符
5. 因此语义上是 move，但性能上未必和 copy 拉开很大差距

这套回答已经够工程和面试使用了。

## 一句话记忆

`std::string` 不是永远“move 就偷指针”；有了 `SSO` 之后，长字符串常常是“转移堆资源”，短字符串常常更像“搬对象内部的小数据”。
