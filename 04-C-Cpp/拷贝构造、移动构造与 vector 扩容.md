# 拷贝构造、移动构造与 vector 扩容

## 为什么这页重要

理解 `vector` 扩容时到底在做什么，绕不开 C++ 的对象语义。`vector` 搬迁元素时，搬的不是“裸字节”这个抽象，而是“对象”。这就直接牵涉到：

- 什么是拷贝构造
- 什么是移动构造
- 为什么移动通常比拷贝便宜
- 为什么元素类型会决定 `vector<T>` 的扩容成本

## 先建立一个最核心的区分

### 拷贝构造

拷贝构造的意思是：

“用一个**已有对象**，构造出一个**新的对象**，并且新对象拥有一份自己的内容。”

典型形式：

```cpp
class A {
public:
    A(const A& other);
};
```

例子：

```cpp
std::string s1 = "hello";
std::string s2 = s1;   // 这里会发生拷贝构造
```

此时：

- `s1` 还在
- `s2` 是新对象
- 两者内容相同
- 但它们是两个独立对象

你可以把它理解成“复印一份”。

### 移动构造

移动构造的意思是：

“用一个**即将不再使用的对象**，构造出一个**新的对象**，然后把原对象内部持有的资源转移给新对象。”

典型形式：

```cpp
class A {
public:
    A(A&& other);
};
```

例子：

```cpp
std::string s1 = "hello";
std::string s2 = std::move(s1);   // 倾向触发移动构造
```

此时：

- `s2` 接管原来 `s1` 的资源
- `s1` 仍然是合法对象
- 但 `s1` 的内容通常处于“有效但未指定”的状态

可以把它理解成“搬家”，不是“复印”。

## 移动构造不等于“浅拷贝”

这是最容易混淆的点。

“移动构造”在效果上**有时看起来像**把几个指针、长度、容量之类的元数据拿过来，但它不应该被简单地等同成“浅拷贝”。

更准确的说法是：

- 浅拷贝：两个对象拷完以后，可能还共同指向同一份资源
- 移动构造：新对象接管资源，旧对象必须被改造成“仍然合法，但不再拥有原资源”的状态

比如一个自己管理堆内存的类：

```cpp
class Buffer {
public:
    Buffer(Buffer&& other) noexcept
        : size_(other.size_), data_(other.data_) {
        other.size_ = 0;
        other.data_ = nullptr;
    }

private:
    size_t size_;
    int* data_;
};
```

这里表面上确实只是把：

- `data_`
- `size_`

这些“元数据”拿过来了。

但关键不是“复制了几个值”，关键是：

- 新对象现在拥有这块资源
- 旧对象已经被清空，不再拥有这块资源

如果你只做“浅拷贝”而不把旧对象置空，那么两个对象会同时指向同一块内存，最后析构时很容易二次释放，这就是错误的。

所以更准确的理解是：

- 拷贝构造：复制资源所有权背后的内容
- 移动构造：转移资源所有权

“指针只是资源句柄，不是资源本身”这句话要记住。

## “构造”是什么意思

这里的“构造”不是赋值。

```cpp
A a;
A b = a;   // 拷贝构造

A c;
c = a;     // 拷贝赋值，不是拷贝构造
```

同理：

```cpp
A b = std::move(a);   // 移动构造

A c;
c = std::move(a);     // 移动赋值
```

所以你要先分清两组概念：

- 拷贝构造 / 移动构造：创建新对象
- 拷贝赋值 / 移动赋值：已有对象之间覆盖内容

## 为什么移动通常更便宜

看一个自己管理堆内存的类：

```cpp
class Buffer {
public:
    Buffer(size_t n) : size_(n), data_(new int[n]) {}

    ~Buffer() { delete[] data_; }

    Buffer(const Buffer& other)
        : size_(other.size_), data_(new int[other.size_]) {
        for (size_t i = 0; i < size_; ++i) {
            data_[i] = other.data_[i];
        }
    }

    Buffer(Buffer&& other) noexcept
        : size_(other.size_), data_(other.data_) {
        other.size_ = 0;
        other.data_ = nullptr;
    }

private:
    size_t size_;
    int* data_;
};
```

### 拷贝构造做了什么

- 重新申请一块新内存
- 把旧数据逐个复制过去

成本通常是 O(n)。

### 移动构造做了什么

- 直接把指针偷过来
- 再把旧对象清空

成本通常接近 O(1)。

这就是为什么“重对象”很依赖移动语义。

## 如果移动构造写得很差，编译器会不会自动把它优化好

通常不要指望。

更准确地说：

- 编译器可能优化掉一些函数调用开销
- 可能做内联、寄存器分配、消除一部分临时量
- 但它一般**不会把“深拷贝式移动构造”自动改写成“偷资源式移动构造”**

原因是这已经不是普通指令级优化，而是：

- 改变对象语义
- 改变资源所有权转移方式

这类事情编译器通常不能擅自替你做。

## 为什么不能乱优化成“偷资源”

看这个例子：

```cpp
class Buffer {
public:
    Buffer(Buffer&& other) {
        data_ = new int[other.size_];
        std::copy(other.data_, other.data_ + other.size_, data_);
        size_ = other.size_;
    }

private:
    int* data_;
    size_t size_;
};
```

这里你明明写的是：

- 新申请内存
- 复制旧数据

这代表你的代码语义就是“构造一个新副本”。  
编译器不能因为它看起来像移动构造，就私自改成：

- `data_ = other.data_`
- `other.data_ = nullptr`

因为那已经不是优化，而是**改程序逻辑**。

## 编译器通常能优化什么

### 1. 内联

如果移动构造很小，编译器可能把函数调用展开掉。

### 2. 消掉一部分临时对象

例如返回值优化、复制消除，这能减少某些构造调用。

### 3. 优化 `memcpy` / `std::copy` 的底层实现

如果元素类型足够简单，底层复制循环可能会被优化得更高效。

但这些优化改变的是：

- 指令数量
- 调用开销
- 局部实现效率

不会改变“你到底是在复制资源，还是在转移资源”这个根问题。

## 什么情况是编译器帮不上忙的

如果你的“移动构造”本质上仍然做了：

- 分配内存
- 复制整块数据
- 复制复杂成员

那它的大头成本仍然在这里。  
编译器最多把周边代码抠得更紧一点，但不会把 O(n) 级的深拷贝神奇变成 O(1) 的偷资源。

## 一个非常重要的区分

### 编译器优化

是在**不改变程序可观察语义**的前提下，把代码跑得更快。

### 你手写高效移动构造

是在**类型设计层面**决定：

- 资源是复制
- 还是转移

后者不是编译器替你兜底的事。

## 一句话总结

编译器也许能把“写得还行的高效移动构造”变得更快，但通常不会把“本质是深拷贝的坏移动构造”自动优化成真正便宜的资源转移。

## `std::move` 到底做了什么

`std::move(x)` **不会真的移动任何东西**。  
它只做一件事：

- 把 `x` 转成一个“可以被移动”的右值表达式

真正的移动是否发生，取决于：

- 类型有没有移动构造
- 当前上下文是否在“用它构造新对象”或“移动赋值”

所以：

```cpp
std::string s1 = "hello";
std::string s2 = std::move(s1);
```

不是 `std::move` 自己把资源搬走了，而是：

- `std::move(s1)` 把 `s1` 变成右值
- `string` 的移动构造函数看到右值，于是执行资源转移

## 它们和 `vector` 扩容怎么衔接

这就是关键连接点。

假设：

```cpp
std::vector<T> v;
```

当 `v` 扩容时，大致过程是：

1. 申请一块更大的新内存
2. 在新内存里重新构造旧元素
3. 销毁旧内存里的旧元素
4. 释放旧内存

第 2 步不是简单的“memcpy 一把梭”，而是：

- 对每个旧元素，调用它的移动构造或拷贝构造

可以粗略理解为：

```cpp
new (new_mem + i) T(std::move(old_mem[i]));
```

如果移动不合适或不可用，则可能退回到拷贝构造。

## 为什么 `int` 看起来像“搬字节”

因为像 `int`、`char`、`double` 这种简单类型没有复杂资源，移动和拷贝的区别几乎看不出来。你把它想成“复制几个字节”问题不大。

但对下面这些类型，就完全不是一个世界了：

- `std::string`
- `std::vector<int>`
- 自己管理文件句柄、socket、堆内存的类
- 带大对象成员的业务类

这类对象扩容搬迁时，代价主要来自对象语义，不是裸内存复制。

## 为什么有时候 `vector` 会拷贝而不是移动

一个常见原因是：

- 类型没有移动构造
- 或移动构造不满足库实现的使用条件

实践里你先抓住这个版本就够了：

- 能安全高效移动时，`vector` 更倾向移动
- 移动不可用时，只能拷贝

所以如果你的类型：

- 只会深拷贝
- 不支持高效移动

那么 `vector<T>` 每次扩容都会很贵。

## 你需要形成的直觉

### 对轻类型

比如：

```cpp
vector<int>
vector<double>
```

扩容成本相对低，因为元素搬迁便宜。

### 对重类型

比如：

```cpp
vector<string>
vector<Buffer>
vector<BigObject>
```

扩容成本可能很高，因为每个元素都要重新构造。

如果 `T` 的移动构造足够便宜，成本会大幅下降；如果只能深拷贝，扩容会很肉眼可见地贵。

## 一个小对比

### 深拷贝思维

“我需要一套新的资源副本给新对象。”

### 移动思维

“旧对象反正快不用了，那我把它手里的资源直接转交给新对象。”

这就是两者最本质的区别：

- 拷贝：复制资源
- 移动：转移资源所有权

## 高频纠偏：拷贝慢，不是因为“它比移动多析构一次”

这句话容易说偏。

更准确地说：

- 拷贝构造会创建一个新对象，并为它准备一套新的资源
- 移动构造也会创建一个新对象，只是它更倾向于接管旧对象手里的资源

所以从对象生命周期看：

- copy 路径会构造新对象，之后旧对象和新对象最终都会各自析构
- move 路径也会构造新对象，之后旧对象和新对象最终也都会各自析构

真正的性能差别，通常不在“有没有析构”，而在：

- copy 往往要额外申请资源、复制内容
- move 往往只需要转交资源控制权，代价更低

对 `std::string` 来说，你可以先这样理解：

- 拷贝：新 `string` 自己再搞一份字符数据
- 移动：新 `string` 把旧 `string` 手里的那份资源接过来，旧对象变成“仍然有效，但值未指定的已移动状态”

但“移动就是改一下指针”也只是一个常见近似图像，不要把它当成铁律：

- 对很多管理堆资源的类型，这个比喻很好用
- 但语言标准只要求“移动语义正确”，不要求实现一定只是改一个指针
- 比如小对象、带内嵌缓冲区的对象，移动时也可能顺手复制一些小数据

所以你现在最该记住的一句是：

- copy 的本质是“复制资源”
- move 的本质是“转移资源控制权”

## 在工程里怎么看一个类型是否适合放进 `vector`

看这几个问题：

1. 这个类型拷贝贵不贵？
2. 这个类型有没有高效移动构造？
3. 扩容是否频繁？
4. 能不能提前 `reserve()`？

如果答案是：

- 拷贝贵
- 移动便宜
- 会频繁扩容

那你就更需要：

- 写好移动构造
- 尽早 `reserve`

## 一句话总结

`vector` 扩容时，搬的是“对象语义”；对简单类型看起来像搬字节，对复杂类型本质是逐个调用移动构造或拷贝构造来重建对象。
