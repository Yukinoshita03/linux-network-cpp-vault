# `noexcept` 与 `vector` 为什么有时宁可拷贝不移动

## `noexcept` 是什么

`noexcept` 的意思是：

- 这个函数承诺“不抛异常”

例如：

```cpp
class Buffer {
public:
    Buffer(Buffer&& other) noexcept;
};
```

这个声明的含义不是“这个函数绝对不会出错”，而是：

- 它承诺不会把异常抛到函数外面
- 如果它内部真的抛了异常，程序通常会直接终止

更精确地说：

- `noexcept` 是对编译器、标准库和调用者的承诺
- 承诺的是“异常不会逃出这个函数边界”
- 不是在说“函数内部绝对不可能出任何问题”

所以 `noexcept` 是一个很强的承诺，不是普通注释。

## 为什么 C++ 这么在意异常

因为很多对象操作不是一步完成的。

比如 `vector` 扩容时，大致要做：

1. 申请新内存
2. 把旧元素搬到新内存
3. 销毁旧元素
4. 释放旧内存

如果第 2 步进行到一半突然抛异常，容器就必须保证自己不要坏掉。

这就引出了一个核心问题：

- 移动构造如果可能抛异常，`vector` 还敢不敢放心用它？

## 为什么 `vector` 有时宁可拷贝不移动

假设 `vector<T>` 扩容时要搬旧元素。

### 情况 1：移动构造是 `noexcept`

那标准库会更放心：

- 直接把旧元素移动到新内存
- 因为它知道这一步不会中途抛异常

这条路径通常更快。

### 情况 2：移动构造可能抛异常

那问题来了：

1. 前面一部分元素已经移动到新内存
2. 后面某个元素移动时抛异常
3. 原来那批元素有些已经被“搬空”
4. 回滚状态就很麻烦

这时如果拷贝构造可用，标准库常常更愿意选拷贝，因为：

- 拷贝不需要破坏原对象
- 即使中途失败，原容器通常更容易保持不变

所以你会看到一个很重要的工程结论：

- **有移动构造还不够**
- **移动构造最好还是 `noexcept`**

## 一个直觉化理解

你可以把 `vector` 想成搬家公司。

### 拷贝

像是：

- 先复印一份旧家具
- 新房里摆好
- 旧房原样不动

如果中途失败，旧房还在。

### 移动

像是：

- 直接把旧房家具搬去新房

这当然更省，但如果搬到一半车翻了，就可能两边都很难处理。

所以如果搬家公司知道：

- 这批家具一定能安全搬完

它就敢直接搬。这个“安全搬完”的承诺，在代码里很多时候就体现为 `noexcept`。

## 为什么资源转移型移动构造通常适合 `noexcept`

比如：

```cpp
class Buffer {
public:
    Buffer(Buffer&& other) noexcept
        : size_(other.size_), data_(other.data_) {
        other.size_ = 0;
        other.data_ = nullptr;
    }

private:
    size_t size_;
    int* data_;
};
```

这类移动构造只是做了：

- 复制几个成员值
- 把旧对象置空

它不需要：

- 重新申请大块内存
- 深拷贝大量数据
- 调复杂逻辑

所以这种移动构造通常很适合标成 `noexcept`。

## 什么时候不该乱加 `noexcept`

如果你的移动构造里其实做了这些事：

- 分配新内存
- 调可能失败的函数
- 做复杂业务逻辑

那它未必真的能保证不抛异常。

这时不能为了“让 `vector` 快一点”就乱标 `noexcept`，否则一旦抛异常，程序会直接终止。

## 你现在该记住的最简版结论

1. `noexcept` = 承诺这个函数不会把异常抛出去
2. `vector` 扩容时很看重异常安全
3. 移动构造如果是 `noexcept`，标准库更敢用移动
4. 移动构造如果可能抛异常，标准库有时会退回去用拷贝

## 一句话总结

`noexcept` 影响的不只是“语法”，它直接影响标准容器在扩容和重排时敢不敢走高效的移动路径。

## `noexcept` 是给编译器看的还是给人看的

答案是：

- 两边都看
- 但首先它是给语言和标准库机制看的，不只是注释

### 给编译器 / 标准库看

因为它会真正影响程序行为：

- 某些优化和选择路径会参考它
- 标准库容器会根据它决定更敢不敢用移动
- 异常传播规则也会因为它改变

所以它不是“写了也无所谓”的文档标签。

### 给人看

它也在表达接口承诺：

- 这个函数设计上应该不会把异常抛出来
- 调用方可以据此理解这个类型的异常安全性质

所以读代码的人看到 `noexcept`，会把它当成 API 语义的一部分。

### 最准确的一句话

`noexcept` 既是给人看的承诺，也是给编译器和标准库看的可执行语义；它不是注释，而是会影响类型行为和容器策略的语言机制。

## `vector` 到底什么时候选移动，什么时候选拷贝

你可以把它理解成标准库内部在扩容时要做一个分支判断。

伪代码可以粗略想成这样：

```cpp
if (T 的移动构造是 noexcept || T 根本不能拷贝) {
    用移动构造把旧元素搬到新内存;
} else {
    用拷贝构造把旧元素复制到新内存;
}
```

很多实现背后的思路接近 `std::move_if_noexcept`：

- 如果“安全移动”，就移动
- 如果“移动可能抛异常且还能拷贝”，就优先拷贝

## 这个判断发生在什么时机

典型场景就是 `vector` 触发扩容时。

比如：

```cpp
std::vector<T> v;
v.push_back(x);   // 如果 size == capacity，这次可能触发扩容
```

一旦扩容，旧内存里的元素要整体迁移到新内存。  
这时候标准库才会面对这个问题：

- 我是该对每个旧元素调用移动构造？
- 还是该调用拷贝构造？

不是你每次写 `std::move` 才触发这个判断，而是容器自己在内部重分配和搬迁元素时会做选择。

## 为什么“移动可能抛异常”会触发改走拷贝

因为标准库想尽量保住一种很强的性质：

- 如果扩容失败，原来的 `vector` 最好保持不变

如果它用的是拷贝：

1. 旧元素还好好待在旧内存
2. 新内存只是多了一些复制出来的新对象
3. 中途失败时，把新内存清理掉即可
4. 原来的 `vector` 通常不受影响

如果它用的是“可能抛异常的移动”：

1. 有些旧元素已经被搬走
2. 那些旧元素可能已经处于“被移动后”的状态
3. 中途再抛异常，旧容器不再是原样

这会让回滚非常棘手。

所以机制不是“库喜欢 copy 不喜欢 move”，而是：

- **能安全 move 就 move**
- **move 不够安全而 copy 又可用时，就宁可 copy**

## 你要形成的画面

扩容时不是一句“搬过去”那么简单，而是两阶段：

1. 先选策略：move 还是 copy
2. 再逐个元素在新内存中重新构造

所以 `noexcept` 的作用不是“让移动更快”，而是：

- 让标准库更敢选“移动”这条路

## 一个很重要的补充

如果类型：

- 不能拷贝
- 只能移动

那标准库很多时候就没有第二条路了，只能移动。  
这时即使移动构造可能抛异常，也可能被迫走移动路径。

所以你可以把选择逻辑记成：

1. 能安全移动：优先移动
2. 移动不够安全但还能拷贝：优先拷贝
3. 根本不能拷贝：那就只能移动
