# `C++ condition_variable` 入门：等待、唤醒与生产者消费者

## 为什么这一页重要

学完：

- `std::thread`
- `mutex`
- `lock_guard`
- `unique_lock`

你已经知道怎么：

- 开线程
- 保护共享数据

但还有一类问题没解决：

- 线程现在不能继续做事时，怎么安全地等
- 另一个线程把条件准备好后，怎么把它叫醒

这就是 `condition_variable` 的位置。

它解决的不是“互斥”，而是：

- **等待某个条件成立，再继续执行**

---

## 先看一个没有它会很别扭的场景

比如有两个线程：

- 生产者：往队列里塞数据
- 消费者：从队列里取数据

问题是：

- 如果队列是空的，消费者现在没法取

这时消费者不能：

1. 不停空转检查
2. 一边占着锁一边死等

因为这样都很蠢。

你真正想要的是：

- 条件不满足时先睡下
- 条件满足时再被唤醒

这就是 `condition_variable` 的语义。

---

## 一句话定义

`condition_variable` 可以先理解成：

- **线程之间的“睡眠 / 唤醒协作器”**

它通常和：

- 一个共享状态
- 一把 `mutex`
- 一个条件判断

配合使用。

---

## 它到底有什么用

如果只说 API，很容易越学越糊。

你可以先只记它解决的这个问题：

- **线程现在做不了事，但它又不该一直占着 CPU 或一直占着锁。**

这就是 `condition_variable` 存在的原因。

比如消费者线程想从队列里取数据：

- 队列里有数据：那就继续取
- 队列里没数据：它现在没法干活

这时它有 3 种选择：

### 选择 1：忙等

```cpp
while (q.empty()) {
}
```

问题是：

- 一直空转
- 白白烧 CPU

### 选择 2：拿着锁死等

```cpp
std::unique_lock<std::mutex> lock(m);
while (q.empty()) {
}
```

问题更大：

- 你把锁一直占着
- 生产者根本进不来塞数据

也就是说：

- 你在等别人把条件改好
- 但你又不让别人进来改

这就僵住了。

### 选择 3：条件变量

正确思路是：

- 队列空了，我现在先睡
- 睡之前把锁让出来
- 等生产者塞完数据后叫醒我
- 我醒来后再重新拿锁，然后继续检查

这就是 `condition_variable` 干的事。

所以你可以把它的作用收成一句特别实用的话：

- **当条件暂时不成立时，让线程体面地等。**

这里的“体面”有两层意思：

- 不空转浪费 CPU
- 不霸着锁挡住别人

---

## 它不是“让线程更安全”，而是“让等待这件事更正确”

`mutex` 解决的是：

- 别同时乱改共享数据

`condition_variable` 解决的是：

- 如果现在条件不满足，线程该怎么等

所以这两个东西解决的根本不是同一类问题。

你甚至可以这样硬记：

- `mutex` 管“别一起上”
- `condition_variable` 管“现在还不能上，那就先睡”

---

## 没有它，你很容易写出两种烂等待

### 烂等待 1：忙等

- 条件没到
- 但线程一直转

这浪费 CPU。

### 烂等待 2：持锁等待

- 条件没到
- 线程一边等一边还占着锁

这会把能改变条件的人也堵死。

而 `condition_variable` 正好卡在这两个坑中间：

- **我不干活时先睡**
- **睡的时候不占锁**
- **条件变化后再被叫醒**

---

## 最小人话模型

你把它想成下面这句就够了：

- **“我现在做不了，先别让我空转；等能做了再叫我。”**

比如消费者对生产者说：

- “队列空了，我先睡，你有货了再叫我”

生产者说：

- “我放进去一个元素了，你可以醒了”

这比你记 `wait/notify` 那堆术语要有用得多。

---

## 它不是拿来保护数据的

这是第一个最容易说混的点。

### `mutex` 干什么

- 保证同一时刻只有一个线程进入临界区

### `condition_variable` 干什么

- 条件不满足时让线程睡眠
- 条件满足后通知它醒来

所以：

- **锁保护共享状态**
- **条件变量协调线程时机**

不要把这两个角色混成一个东西。

---

## 最小直觉模型

你可以先把它想成这套对话：

消费者说：

- “队列现在是空的，我先睡”

生产者说：

- “我已经塞进数据了，叫醒你”

这对应的就是：

- `wait()`
- `notify_one()` / `notify_all()`

---

## 最小例子

```cpp
#include <condition_variable>
#include <iostream>
#include <mutex>
#include <thread>

std::mutex m;
std::condition_variable cv;
bool ready = false;

void worker() {
    std::unique_lock<std::mutex> lock(m);
    cv.wait(lock, [] { return ready; });
    std::cout << "worker start\n";
}

int main() {
    std::thread t(worker);

    {
        std::lock_guard<std::mutex> lock(m);
        ready = true;
    }

    cv.notify_one();
    t.join();
}
```

这个例子可以先这样读：

1. `worker` 先拿锁
2. 检查 `ready`
3. 如果 `ready == false`，就睡眠
4. 主线程把 `ready` 改成 `true`
5. 主线程发通知
6. `worker` 被唤醒，重新继续

---

## `wait()` 到底做了什么

这是最核心的一层。

`wait()` 不是“拿着锁傻等”。

它做的是一整套动作：

1. 当前线程先持有锁
2. 检查条件
3. 如果条件不满足，**原子地释放锁并睡眠**
4. 被唤醒后，**先重新拿回锁**
5. 再次检查条件
6. 条件满足才继续往下跑

重点就是这句：

- **释放锁并睡眠**

因为如果等待线程睡着时还一直拿着锁，
那生产者就没法进来改条件了。

### 一个最容易说反的点

`cv.wait(lock)` **不是**：

- “我还占着这把锁，只是睡着了”

而是：

- **我先把这把锁交出去，然后自己睡着**

否则就会出现死局：

1. 你在等 `ready` 变成 `true`
2. 但你睡着时还一直拿着锁
3. 那别人就进不来修改 `ready`
4. 于是你等的条件永远不可能成立

所以 `condition_variable` 的关键设计恰恰是：

- **等待时不持锁**
- **醒来后再重新拿锁**

---

## `cv.wait(lock)` 不是“只等别人释放锁”

如果你把：

```cpp
cv.wait(lock);
```

理解成：

- “别的线程把锁放了，我再去抢”

这 **不完全对**。

更准确地说，它分两段：

1. 当前线程先拿着 `lock`
2. 调用 `wait(lock)` 时，**释放这把锁并睡眠**
3. 等待别人 `notify_one()` / `notify_all()`，或者发生假唤醒
4. 醒来后，**再去重新竞争这把锁**
5. 只有重新拿到锁以后，`wait()` 才会返回

所以它等的不是“锁什么时候空出来”这么简单，
而是：

- **先等唤醒事件**
- **再等重新拿到锁**

这和普通的“锁阻塞”不是一回事。

### 普通锁阻塞是什么

比如：

```cpp
std::mutex m;
m.lock();

// 别的线程这里调用 m.lock()，它是在等这把锁被释放
```

这里线程阻塞的原因是：

- **锁当前被别人占着**

而 `cv.wait(lock)` 不是这样。

它不是“因为拿不到锁才卡住”，
而是：

- **自己主动把锁放掉**
- **挂到 condition variable 上睡眠**
- **被唤醒后再回来拿锁**

### 一个最关键的纠偏

就算锁现在已经空了，
如果没人 `notify`，`cv.wait(lock)` 也不会因为“锁可用了”就自己返回。

所以：

- 它不是单纯等锁
- 它本质上是在等“被唤醒后再重新拿锁”

### 为什么裸 `wait(lock)` 通常还要配 `while`

因为裸 `wait(lock)` 根本不知道你到底在等什么条件，
所以常见正确写法是：

```cpp
std::unique_lock<std::mutex> lock(m);
while (!ready) {
    cv.wait(lock);
}
```

这表示：

- 醒了以后别急着往下跑
- 先重新拿锁
- 再检查 `ready` 是不是真的满足

而更推荐的缩写写法就是：

```cpp
cv.wait(lock, [] { return ready; });
```

### 醒来后到底是怎么“重新抢锁”的

这个“抢”本质上不是 `condition_variable` 自己替你把锁拿回来，
而是：

- **`wait()` 内部会重新走一遍这把 `mutex` 的加锁路径**

你可以把它粗略理解成这种伪代码：

```cpp
void wait(unique_lock<mutex>& lock) {
    // 1. 原子地：把当前线程挂到 cv 的等待队列，并释放 mutex
    unlock(lock);
    sleep_on_cv();

    // 2. 被 notify 或假唤醒后，线程变成可运行
    // 3. 然后重新对这把 mutex 执行 lock()
    lock.lock();

    // 4. 只有这里成功后，wait() 才真正返回
}
```

所以“抢锁”其实就是：

- 醒来之后，再对同一把 `mutex` 做一次 `lock()`

如果此时锁已经空了，它就拿到锁继续走。

如果此时锁还被别的线程拿着，
那它会：

- 卡在这次重新 `lock()` 上
- 等那把锁真正释放
- 拿到锁以后才从 `wait()` 返回

### 也就是说，线程可能已经醒了，但 `wait()` 还没返回

这是很重要的一层。

很多人会误以为：

- `notify_one()` 一发，对方线程就立刻继续执行

其实不是。

更准确地说：

1. `notify_one()` 只是让等待线程从“睡在 cv 上”变成“可以竞争执行”
2. 等待线程醒来后，还得重新竞争 `mutex`
3. 只有 `mutex` 真拿到了，它才真的继续往下跑

所以：

- **被唤醒 != 已经继续执行**
- **被唤醒只是获得了“回来重新抢锁”的资格**

### 一个最直观的时序

```cpp
// 线程 A
std::unique_lock<std::mutex> lock(m);
cv.wait(lock);
// 只有重新拿到 m 之后，才会运行到这里

// 线程 B
std::lock_guard<std::mutex> lock(m);
ready = true;
cv.notify_one();
// 注意：这里线程 B 如果还没离开作用域，m 还在它手里
```

这时候会发生：

1. 线程 A 在 `wait()` 里睡着，并且已经把 `m` 释放了
2. 线程 B 拿到 `m`，修改 `ready`
3. 线程 B 调用 `notify_one()`
4. 线程 A 被唤醒，但这时 **不代表它马上从 `wait()` 返回**
5. 如果线程 B 还没释放 `m`，线程 A 还得继续等
6. 直到线程 B 解锁，线程 A 才能重新拿到 `m`
7. 线程 A 这时才真正从 `wait()` 返回

所以你可以把它收成一句话：

- **`notify` 负责叫醒，`mutex` 负责卡住谁先继续。**

---

## 高频纠偏：不用 `wait()`，不也一样会“卡住”吗

这是新手最容易冒出来的疑问。

比如你会想：

```cpp
void worker() {
    std::unique_lock<std::mutex> lock(m);
    // 不写 wait，线程不也会停在这里吗？
}
```

或者：

```cpp
void worker() {
    std::unique_lock<std::mutex> lock(m);
    while (!ready) {
    }
}
```

表面看它们都“卡住”了，  
但和 `wait()` 的差别是巨大的。

### 情况 1：只是拿着锁不继续

如果线程拿到锁后不走 `wait()`，而是就这么卡着，
那它会：

- 一直占着这把锁

结果就是：

- 其他线程根本进不来改 `ready`
- 条件永远也没法从 `false` 变成 `true`

这不是正常等待，
这是把门锁上以后自己堵在门口。

### 情况 2：忙等

如果你写：

```cpp
while (!ready) {
}
```

那线程虽然在“等”，
但它：

- 不停占 CPU 空转
- 而且如果外面还拿着锁，就更麻烦

这叫忙等，不叫条件等待。

### `wait()` 真正多做了哪一步

关键就在这里：

- **它不是单纯阻塞**
- **它是“释放锁以后再阻塞”**

这才是本质区别。

所以你要把两种“卡住”分清：

1. **锁阻塞**
   - 线程因为拿不到锁被卡住
2. **条件等待**
   - 线程发现条件不满足，主动释放锁并睡眠，等别人把条件改好再叫醒

这两种不是一个层面的东西。

一句话记忆：

- 不用 `wait()` 的“卡住”，常常是在挡别人的路
- `wait()` 的“卡住”，是在主动把路让出来再去睡

---

## 为什么它必须配 `unique_lock`

因为 `wait()` 这个过程要求锁对象支持：

- 现在我先 `unlock`
- 醒来之后我再 `lock`

而：

- `lock_guard` 只适合“构造就锁，析构就解”
- 它不给你显式控制 `unlock()` / `lock()` 的这套能力

所以条件变量通常写成：

```cpp
std::unique_lock<std::mutex> lock(m);
cv.wait(lock, [] { return ready; });
```

不是因为 `unique_lock` 更高级，
而是因为 `wait()` 的机制天然需要它这种可控锁包装器。

### 但更本质的一层：为什么 `wait` 非得传这把锁

这不只是因为 API 想让你传。

更本质的原因是：

- **等待线程要在“检查条件”和“进入睡眠”之间，和修改条件的线程做正确同步。**

如果没有这把锁，就会出很经典的问题：

- **丢唤醒**

### 什么叫丢唤醒

看这个危险时序：

1. 线程 A 想等 `ready == true`
2. 线程 A 先检查一眼：`ready == false`
3. 这时线程 A 还没真正睡下去
4. 线程 B 把 `ready = true`，然后 `notify_one()`
5. 线程 A 这时才开始睡

结果就尴尬了：

- 通知已经发过了
- 但线程 A 睡晚了，没接到这次通知
- 它可能就一直睡下去

这就叫：

- **条件已经变好了，但等待线程错过了那次通知**

### 传锁的作用，就是把这段时序焊死

正确做法里：

```cpp
std::unique_lock<std::mutex> lock(m);
cv.wait(lock, [] { return ready; });
```

背后的关键不是“有个锁看起来更正规”，
而是：

1. 等待线程先拿锁
2. 在锁保护下检查 `ready`
3. 如果不满足，`wait()` 会 **原子地** 做两件事：
   - 把当前线程挂到条件变量等待队列
   - 释放这把锁

这样线程 B 就不可能在“你刚检查完条件，但还没真正进入等待”这个缝里插进来把通知丢掉。

你可以把它理解成：

- **锁把“我确认现在还不能做”和“那我正式去睡了”这两步粘成一个整体。**

### 还有另一层：条件本身通常也是共享数据

比如：

- `ready`
- `q.empty()`
- `count > 0`

这些条件本身通常就来自共享状态。

既然是共享状态，
那检查它的时候本来就应该加锁。

所以传锁给 `wait()` 还有第二层含义：

- **我检查条件时用的是哪把锁，等待时也必须交给同一把锁配合。**

这样“检查条件的人”和“修改条件的人”看的是同一份受保护状态。

### 一句话收住

`wait` 要传锁，不只是为了以后能 `unlock/lock`，
更是为了保证：

- **检查条件**
- **进入等待**
- **别人修改条件并通知**

这三件事之间没有竞态缝隙。

---

## 为什么 `wait(lock, pred)` 比裸 `wait(lock)` 更推荐

推荐写法：

```cpp
cv.wait(lock, [] { return ready; });
```

它等价于一个循环检查：

```cpp
while (!ready) {
    cv.wait(lock);
}
```

这背后有两个原因。

### 1. 可能被“假唤醒”

也就是说：

- 线程醒了
- 但条件不一定真的满足

所以醒来之后不能直接往下跑，
还得再检查一次条件。

### 2. 可能被别人抢先消费掉条件

比如多个消费者都在等：

- 你被叫醒了
- 但另一线程先一步拿走了资源

所以你醒来后也必须重新检查条件。

因此你要记死一句：

- **条件变量等的不是“通知”，等的是“条件”**

通知只是“你可以起来再看看了”，
不是“条件百分百已经满足”。

### 你现在这两句该怎么说得更准

如果你想自己口述，可以收成这样：

1. `wait(lock, pred)` 返回时还要把锁拿在手里，
   不是单纯为了“再决定要不要睡”，
   而是为了让线程在 **条件已经成立** 之后，
   能立刻在同一把锁保护下安全地继续访问和修改共享状态。
2. `while` 也不是“轮询”那个意思，
   因为线程并不是一直空转检查。
   更准确地说，它是：
   **每次被唤醒并重新拿到锁后，再循环复查一次条件。**

所以第 2 点最好别说成：

- “因为它是一个轮询过程”

而更该说成：

- **“因为可能假唤醒，也可能条件被别的线程先改掉，所以每次醒来后都要重新检查条件。”**

---

## `cv.wait(lock, [] { return ready; });` 里这个 lambda 到底干嘛

这个 lambda 干的事非常具体：

- **充当“条件检查函数”**

也就是说，标准库会反复问它：

- “现在条件满足了吗？”

这里：

```cpp
[] { return ready; }
```

意思就是：

- 不捕获任何局部变量
- 每次被调用时，返回当前 `ready` 的值

所以这句代码可以先翻译成人话：

- “如果 `ready` 还没变成 `true`，那我就继续等；只有 `ready == true` 才真的继续往下执行。”

### 它不是拿来“通知”的

要分清两件事：

- `notify_one()` 负责叫醒等待线程
- 这个 lambda 负责检查“现在是不是真的可以继续了”

所以：

- `notify` 是喊你起来看看
- `lambda` 是你起来之后自己确认条件

### 它近似等价于什么

这句：

```cpp
cv.wait(lock, [] { return ready; });
```

近似等价于：

```cpp
while (!ready) {
    cv.wait(lock);
}
```

所以这个 lambda 本质上就是把：

- “循环检查条件”

这一层逻辑塞进了 `wait()` 的第二个参数里。

### 为什么这里 `[]` 里面什么都不写，也能访问 `ready`

因为这里的 `ready` 是全局变量。

全局变量不是 lambda 的局部捕获对象，
所以即使写空捕获列表：

```cpp
[]
```

也照样能访问它。

如果 `ready` 不是全局变量，而是外层局部变量，
那就通常要写成类似：

```cpp
[&] { return ready; }
```

或者：

```cpp
[&ready] { return ready; }
```

### 你现在最该记住的一句

`wait(lock, pred)` 里的 `pred`，就是：

- **“我到底在等什么条件成立”的那句判断**

### 可以把整句收成什么规则

你现在可以把：

```cpp
cv.wait(lock, pred);
```

收成这句非常实用的话：

- **执行到这里时，线程必须先处在“拿着这把锁”的状态；而真正继续往下执行之前，`pred()` 也必须判成 `true`。**

拆开就是：

1. 进入 `wait(lock, pred)` 前，当前线程先持有 `lock`
2. 如果 `pred()` 已经是 `true`，那就不睡，直接继续
3. 如果 `pred()` 是 `false`，那就释放锁并睡眠
4. 被唤醒后重新拿锁
5. 再检查 `pred()`
6. 只有 `pred()` 终于为 `true`，才真的离开 `wait()` 继续往下跑

所以你这句理解基本是对的，但最好说得再精确一点：

- 不是“执行到这里必须拿到锁，而且必须立刻 true”
- 而是“进入等待时先持锁；离开等待继续执行时，条件必须为 true”

### 它是不是“线程苏醒后调用的那个函数”

可以先说：

- **很接近，但不够准确**

更准确地说，`pred` 是：

- **每次等待线程需要判断“现在能不能继续”时调用的条件函数**

也就是说，它不只是“醒来后调用一次”，而是可能在这些时机被检查：

1. 刚进入 `wait(lock, pred)` 时
2. 每次被唤醒并重新拿到锁之后
3. 如果条件还不满足，就继续睡；下次醒来再检查

所以它更像：

- “醒来后要重新确认条件的判断函数”

而不是：

- “一被唤醒就自动执行一次的回调函数”

你可以继续把它记成：

```cpp
while (!pred()) {
    cv.wait(lock);
}
```

这里就能看出来：

- 真正负责唤醒的是 `notify_one()` / `notify_all()`
- 真正负责决定“这次醒来能不能往下走”的是 `pred()`

### 一句最短纠偏

`cv.wait(lock, pred)` **不是**：

- “放锁睡觉，醒来后调用 `pred()` 一次”

而更接近：

```cpp
while (!pred()) {
    cv.wait(lock);
}
```

所以第二个参数 `pred` 的角色是：

- **条件判断函数**

不是：

- **唤醒后的回调函数**

它可能在这些时机被检查：

1. 刚进入 `wait(lock, pred)` 时
2. 每次被唤醒并重新拿到锁以后
3. 如果条件还不成立，就继续等，下次再检查

因此更准确的人话是：

- **先在锁保护下反复判断条件；条件不成立就放锁睡觉；被唤醒并重新拿到锁后，再判断能不能继续。**

### 再补一条最容易说错的口述

如果把 `pred()` 粗暴看成返回 `0 / 非 0`：

- `pred() == 0`：条件不成立，`wait` 会去放锁并等待
- `pred() != 0`：条件成立，`wait` 不再继续睡，而是直接往下走

注意这里最容易说反的一点：

- **不是“返回非 0 就放锁”**
- 而是“返回 0 才继续等；返回非 0 就结束等待”

并且 `wait(lock, pred)` 真正返回时：

- **这把锁已经重新拿在当前线程手里了**

---

## `notify_one()` 和 `notify_all()` 有什么区别

### `notify_one()`

- 唤醒一个等待中的线程

适合：

- 一个资源来了，只需要一个线程处理

### `notify_all()`

- 唤醒所有等待中的线程

适合：

- 状态变化可能让很多线程都能继续
- 或者你懒得精细挑一个

但注意：

- 被唤醒不等于都能继续执行
- 它们醒来后还要重新抢锁、重新检查条件

所以 `notify_all()` 不是“所有人都一起干活”，
而是“所有人都起来重新判断一遍”。

## 经典生产者消费者模型

这是条件变量最标准的落地。

```cpp
#include <condition_variable>
#include <iostream>
#include <mutex>
#include <queue>
#include <thread>

std::mutex m;
std::condition_variable cv;
std::queue<int> q;

void producer() {
    for (int i = 1; i <= 5; ++i) {
        {
            std::lock_guard<std::mutex> lock(m);
            q.push(i);
        }
        cv.notify_one();
    }
}

void consumer() {
    for (int i = 1; i <= 5; ++i) {
        std::unique_lock<std::mutex> lock(m);
        cv.wait(lock, [] { return !q.empty(); });

        int x = q.front();
        q.pop();
        lock.unlock();

        std::cout << "consume: " << x << "\n";
    }
}
```

这里最重要的不是语法，而是模型：

1. 队列是共享状态
2. 队列要用锁保护
3. 消费者不是空转检查，而是在队列空时睡眠
4. 生产者塞入数据后发通知

这就是“互斥 + 等条件”的配合。

---

## 为什么通知前后常看到“先改状态，再 notify”

这是一个很重要的写法习惯。

通常会这样：

```cpp
{
    std::lock_guard<std::mutex> lock(m);
    ready = true;
}
cv.notify_one();
```

也就是：

1. 先在锁保护下修改共享状态
2. 再发通知

这样等待线程被唤醒时，
看到的是已经更新好的状态。

你现在先记这个工程直觉就够了：

- **状态先改对，再通知别人来看**

---

## 这和“忙等”有什么区别

错误思路常常是：

```cpp
while (!ready) {
}
```

这叫忙等。

问题是：

- 会一直占 CPU
- 很浪费

而 `condition_variable` 的思路是：

- 条件不满足就睡
- 需要时再醒

所以它比忙等更像真正的线程协作。

---

## 新手最容易犯的错

### 1. 只等通知，不等条件

比如：

```cpp
cv.wait(lock);
// 醒了就直接往下做
```

这很危险。

你应该默认：

- 醒来后还得重新检查条件

所以优先写：

```cpp
cv.wait(lock, pred);
```

### 2. 忘了共享状态也要加锁

条件变量不是替代锁的。

这些共享状态：

- `ready`
- `queue`
- `count`

本身还是要被 `mutex` 保护。

### 3. 拿着锁做太多慢操作

比如：

- 打印
- 文件 I/O
- 网络 I/O

如果这些和共享状态无关，
就尽量别长时间放在锁里。

### 4. 以为 `notify_one()` 会把条件变成真

不会。

`notify_one()` 只是：

- 通知一个等待线程起来看看

真正让条件成立的是：

- 你对共享状态的修改

---

## 你现在最该记住的 4 句话

1. `mutex` 保护共享状态，`condition_variable` 协调等待时机。
2. 线程等待的不是“通知”，而是“条件成立”。
3. `wait()` 会在睡眠时临时释放锁，醒来后重新拿锁。
4. 条件变量几乎总是和 `unique_lock` + 条件判断一起出现。

---

## 面试够用回答

如果别人问：

“`condition_variable` 是干嘛的？”

你可以这样答：

`condition_variable` 用来做线程之间的等待和唤醒协作。  
当某个条件还不满足时，线程可以在持有 `unique_lock` 的情况下调用 `wait()` 进入睡眠；`wait()` 会先释放锁，等被通知后再重新拿锁并继续检查条件。它通常和 `mutex` 一起用，典型场景是生产者消费者模型。

如果再追问：

“为什么要配 `unique_lock`？”

你就答：

因为 `wait()` 的机制要求等待线程在睡眠前释放锁、醒来后重新加锁，所以需要一个支持受控 `unlock/lock` 的锁包装器，`lock_guard` 太死，`unique_lock` 正好适合。

---

## 一句话记忆

`condition_variable` 不是拿来保护数据的，而是让线程在“条件没满足时睡下，条件满足后再醒来继续”。
