# C++ 线程深入理解：执行流、共享内存与生命周期

## 先给一句话定义

线程不是“一个函数开出去跑”这么简单。

更准确地说：

- **线程是进程内部一个可被调度的执行上下文**

这里面有两个词最关键：

1. **执行上下文**
2. **可被调度**

---

## 什么叫“执行上下文”

一个线程之所以能继续往下跑，不是因为它“记得自己是谁”，而是因为系统替它保存了一套运行状态。

你可以先把一个线程最低限度理解成这几样东西：

1. 当前执行到哪条指令了
2. 当前寄存器里装着什么值
3. 它自己的调用栈长什么样
4. 它现在是正在运行、就绪、阻塞，还是结束了

所以线程不是“代码”本身。

代码是大家共享的一份机器指令；
线程更像是：

- **同一份代码上的一个独立运行轨迹**

---

## 进程和线程到底谁管什么

这块如果不想清楚，后面所有并发概念都会飘。

### 进程更像资源边界

一个进程通常拥有：

1. 自己的虚拟地址空间
2. 代码段
3. 全局/静态区
4. 堆
5. 打开的文件描述符
6. 各类 OS 资源

### 线程更像执行边界

同一进程里的线程通常共享：

1. 代码段
2. 堆
3. 全局变量
4. 打开的文件描述符

但每个线程自己有：

1. 寄存器上下文
2. 调用栈
3. 调度状态
4. 线程 ID

所以最值得记住的一句话是：

- **进程负责“我拥有什么资源”，线程负责“谁在执行这些资源上的代码”**

---

## 一个最重要的图

```text
Process
├─ code/text        <- 线程共享
├─ global/static    <- 线程共享
├─ heap             <- 线程共享
├─ fd/socket        <- 线程共享
├─ thread A
│  ├─ registers
│  ├─ PC
│  └─ stack         <- A 私有
└─ thread B
   ├─ registers
   ├─ PC
   └─ stack         <- B 私有
```

这个图一旦吃透，很多现象就自然了：

1. 为什么全局变量会有竞态
2. 为什么堆对象会被多个线程一起改
3. 为什么局部变量默认是线程私有的
4. 为什么把局部变量的引用/指针传给别的线程会危险

---

## 一个高频纠偏：哪些是线程共享，哪些是线程独享

这块非常容易说反，最常见的正确版本是：

### 同一进程内，线程共享的东西

1. 地址空间
2. 代码段
3. 全局/静态区
4. 堆
5. 文件描述符 / socket

### 同一进程内，线程独享的东西

1. 寄存器上下文
2. 程序计数器
3. 调用栈
4. 调度状态
5. 线程 ID

所以：

- **不是“所有进程共享堆、代码、全局变量”**
- 而是“同一个进程里的多个线程共享这些资源”

不同进程默认有各自独立的地址空间。

也不是：

- **线程独享地址空间、代码段、堆、全局区、fd**

这些恰恰通常是同进程线程共享的部分。

线程真正独享的，主要是：

- **寄存器 + 栈 + 调度上下文**

这句一定要背稳。

---

## 为什么“局部变量是私有的”还会出事

因为“局部变量在谁的栈上”是一个问题，
“别人能不能拿到这个地址”是另一个问题。

比如：

```cpp
void f() {
    int x = 10;
    std::thread t([&x] {
        std::cout << x << "\n";
    });
    t.detach();
}
```

这里 `x` 本来在当前线程自己的栈上，是私有的。

但你把 `x` 的引用交给了另一个线程，于是：

- 别的线程现在也能碰这块内存了

更危险的是：

- `f()` 返回后，这个栈变量可能已经死了
- 被 `detach` 的线程却可能还在用它

所以“线程私有栈”不等于“天然安全”。

真正决定安不安全的是：

- **这块数据有没有被别的线程共享出去**

### 共享出去之后，问题分成哪两类

这块很容易混。

把局部变量共享给别的线程之后，问题不只是一个“竞争”。

更准确地说，通常会分成两层：

1. **生命周期问题**
2. **同步问题**

### 生命周期问题

如果原来的局部变量先死了，但别的线程还在用它，就会出现：

- 悬空引用
- 悬空指针

比如：

```cpp
void f() {
    int x = 10;
    std::thread t([&x] {
        std::cout << x << "\n";
    });
    t.detach();
}
```

这里最先爆出来的核心问题是：

- `f()` 结束后，`x` 的生命周期结束
- 后台线程却可能还拿着 `x` 的引用

这首先是：

- **生命周期失控**

### 同步问题

如果局部变量虽然还活着，但已经被多个线程共享访问，且至少有一个线程在写，就会进入：

- data race
- 竞态条件

比如：

```cpp
int x = 0;

std::thread t([&x] {
    ++x;
});

++x;
t.join();
```

这里 `x` 还活着，
但两个线程在并发修改它，
而且没有同步。

这时候核心问题是：

- **共享可变状态没有被正确同步**

### 两类问题可能同时存在

最容易踩坑的是：

- 你既没有管好生命周期
- 也没有管好同步

所以更准确的心智模型应该是：

- **局部变量默认在线程私有栈上，通常安全**
- **一旦被共享出去，就不再只是“局部变量”这个问题，而要同时检查生命周期和同步**

---

## `std::thread` 到底做了什么

从 C++ 视角看：

```cpp
std::thread t(f, arg1, arg2);
```

你看到的是一句构造。

但底层可以先理解成 4 步：

1. 把 `f` 和参数保存起来
2. 请求 OS 创建一个新线程
3. 新线程启动后，执行一个库内部的包装入口
4. 包装入口再去真正调用 `f(arg1, arg2)`

所以 `std::thread` 做的不是“直接运行你的函数”，而更像：

- **把一个可调用对象和参数封装起来，交给 OS 启动的新执行流去调用**

这也解释了为什么：

- 参数个数/类型必须能组成一次合法调用
- 默认常常是按值保存参数
- 引用语义常常要靠 `std::ref`

---

## `std::thread` 对象和线程本身不是一回事

这个点非常重要。

很多人会下意识以为：

- `std::thread` 对象 = 线程

其实更准确地说：

- **`std::thread` 对象是你在 C++ 里握着的那个“线程句柄”**

线程本身是 OS 调度的执行实体；
`std::thread t` 只是当前线程里一个 C++ 对象，用来管理它。

所以会出现这些现象：

1. 线程可能已经跑完了，但 `t` 还在
2. `t` 被 move 走了，线程还活着
3. `t.detach()` 后对象不再管理它，但线程还继续跑

这个视角一对，`join/detach` 就好理解了。

---

## `join` 到底是什么语义

`join()` 不等于“创建依赖关系”。

它更准确的意思是：

- **当前线程等待目标线程结束**

所以：

```cpp
t.join();
```

本质上是在说：

- 我现在先不往下走
- 等 `t` 代表的那个线程彻底结束

这解决的是两个问题：

1. 生命周期收口
2. 结果同步点

### 为什么说它还是“生命周期收口”

因为一个 joinable 的 `std::thread` 对象如果直接析构，会触发：

- `std::terminate()`

标准库这样设计，就是逼你明确表态：

1. 要么 `join`
2. 要么 `detach`

不能装作没看见。

---

## `detach` 到底危险在哪

`detach()` 的语义是：

- **把线程放到后台继续运行，并放弃之后的显式管理**

调用之后：

1. 你不能再 `join`
2. 这个线程什么时候结束，你不再直接收口
3. 它访问的数据生命周期必须你自己保证

所以 `detach` 真正危险的不是“语法难”，而是：

- **生命周期和所有权边界一下子变松了**

最常见的坑就是：

1. 线程里引用了局部变量
2. 外层函数先返回
3. 后台线程还没跑完
4. 悬空引用 / 悬空指针

所以初学阶段的默认策略仍然应该是：

- **先 `join`，谨慎 `detach`**

---

## 并发和并行不是一回事

这点也要分清。

### 并发

- 多个执行流在时间上交错推进

哪怕只有一个 CPU 核，也能并发。

因为 OS 可以不停切换：

- 先跑一会线程 A
- 再跑一会线程 B

### 并行

- 多个执行流在同一时刻真的同时跑

这通常需要多个核心。

所以：

- **并发强调结构上有多条执行流**
- **并行强调物理上真的同时执行**

---

## 调度器眼里的线程

站在 OS 角度，线程通常会在这些状态之间切换：

1. 就绪
2. 运行
3. 阻塞
4. 结束

比如：

- 没抢到 CPU：就绪
- 正在占用 CPU：运行
- 等锁、等 IO、等条件：阻塞

这意味着你的程序顺序在并发里不是线性的，而是：

- **随时可能在任意一个点被切走**

这就是为什么多线程 bug 这么讨厌：

- 你写的是一条逻辑线
- 机器跑的是交错执行

---

## 为什么竞态不是“结果偶尔不对”那么简单

在 C++ 里，两个线程如果对同一份内存做了冲突访问，且没有正确同步，这叫：

- **data race**

它比“得到一个意外数字”更严重。

因为标准层面这属于：

- **未定义行为**

也就是说，问题不只是“最后值不对”，而是：

1. 编译器可以做你意想不到的优化
2. CPU 可以出现你没预料的可见性现象
3. 程序行为不再有语言层保证

所以并发里最该怕的，不是“运气不好”，而是：

- **你已经跑出了语言保证的边界**

---

## `mutex` 真正保护的不是“变量”，而是“不变量”

这是比“同一时刻只有一个线程进来”更深一层的理解。

比如一个队列对象，真正要保护的不是某个单独成员变量，而是：

- 队列内部结构始终保持一致

再比如：

```cpp
balance -= 100;
stock += 1;
```

你要保护的不是单独的 `balance` 或 `stock`，
而是这两个动作组成的业务不变量：

- 扣钱和加库存必须一起完成

所以锁保护的真正对象应该理解成：

- **一段必须原子地维护对象一致性的临界区**

这就是为什么我前面一直强调：

- 锁得太小不行
- 锁得太大也不行

---

## `lock_guard` 为什么是现代 C++ 的默认入口

因为锁本质上也是资源。

你一旦手写：

```cpp
m.lock();
// ...
m.unlock();
```

就把自己暴露给这些风险：

1. 中途 `return`
2. 中途抛异常
3. 后续代码改复杂后忘记解锁

而 `lock_guard` 是 RAII：

- 构造时拿锁
- 析构时放锁

所以它利用的是你已经熟悉的对象生命周期模型，来管理并发资源。

这也是现代 C++ 很常见的一条主线：

- **把危险资源交给对象生命周期托管**

---

## 你现在该建立的线程心智模型

如果把前面的内容收成一套模型，应该是：

1. 线程是进程内一个可调度执行流
2. 同进程线程共享大部分内存和资源
3. 每个线程有自己的栈和寄存器上下文
4. `std::thread` 是线程句柄，不是线程本体
5. `join` 是显式收口，`detach` 是放弃后续管理
6. 一旦共享可变状态，就必须讨论同步
7. `mutex` 管的不是语法，而是对象/业务不变量

如果这 7 句你真的能自己讲出来，线程这一章就不再是“API 记忆”了。
