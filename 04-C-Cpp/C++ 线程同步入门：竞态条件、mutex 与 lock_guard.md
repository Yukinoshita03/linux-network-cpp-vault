# C++ 线程同步入门：竞态条件、mutex 与 lock_guard

## 先给一句话定义

这一页真正要讲清的是：

- **多个线程一旦共享可变数据，正确性就不再只取决于代码，还取决于执行时序**

这就是线程最核心的难点。

---

## 什么是竞态条件

竞态条件可以先理解成：

- **结果依赖“谁先执行、谁后执行”的并发错误**

它常见于这种场景：

1. 多个线程
2. 同时访问同一份数据
3. 至少有一个线程在修改
4. 中间没有正确同步

一旦满足这 4 条，就该警觉了。

---

## 为什么 `x++` 都可能出问题

很多新手最容易低估这一点：

- `x++` 看起来像一个动作
- 但底层不是“瞬间完成”

你可以先把它粗暴理解成 3 步：

1. 读出 `x`
2. 加一
3. 写回 `x`

如果两个线程交错执行，就可能发生覆盖。

比如：

```cpp
int x = 0;

// 线程 A
// 读到 x = 0
// 加一，准备写回 1

// 线程 B
// 也读到 x = 0
// 加一，准备写回 1
```

最后两边都写回 `1`，结果不是你直觉里的 `2`。

所以这里的关键不是“线程很多”，而是：

- **读-改-写被别的线程插进来了**

---

## 最小坏例子

```cpp
#include <iostream>
#include <thread>

int x = 0;

void work() {
    for (int i = 0; i < 100000; ++i) {
        x++;
    }
}

int main() {
    std::thread t1(work);
    std::thread t2(work);

    t1.join();
    t2.join();

    std::cout << x << "\n";
}
```

你可能以为一定输出 `200000`，但并不保证。

因为：

- 两个线程都在无保护地修改同一个全局变量 `x`

这就是最典型的数据竞争。

---

## `mutex` 到底在干嘛

`mutex` 可以先理解成：

- **同一时刻只允许一个线程进入临界区**

这里的“临界区”就是：

- 那段访问共享数据、必须串行执行的代码

所以 `mutex` 不是“让线程更快”，而是：

- **让共享状态的修改不乱**

---

## 加锁后的版本

```cpp
#include <iostream>
#include <mutex>
#include <thread>

int x = 0;
std::mutex m;

void work() {
    for (int i = 0; i < 100000; ++i) {
        m.lock();
        x++;
        m.unlock();
    }
}

int main() {
    std::thread t1(work);
    std::thread t2(work);

    t1.join();
    t2.join();

    std::cout << x << "\n";
}
```

这时候你可以先直觉理解成：

- 谁先拿到锁，谁先改
- 没拿到锁的线程先等着

所以 `x++` 那 3 步被包成了“同一时刻只能一个人做完”。

---

## 但手写 `lock()` / `unlock()` 很危险

因为一旦你中途 `return`，或者以后代码变复杂抛了异常，就可能：

- `lock()` 了
- 但忘了 `unlock()`

这会直接把别的线程全卡住。

所以现代 C++ 更常见的写法是：

- `std::lock_guard`

---

## `lock_guard` 的语义

`lock_guard` 可以先理解成：

- **构造时加锁，析构时自动解锁**

也就是把“锁的生命周期”绑到一个局部对象身上。

```cpp
#include <iostream>
#include <mutex>
#include <thread>

int x = 0;
std::mutex m;

void work() {
    for (int i = 0; i < 100000; ++i) {
        std::lock_guard<std::mutex> lock(m);
        x++;
    }
}

int main() {
    std::thread t1(work);
    std::thread t2(work);

    t1.join();
    t2.join();

    std::cout << x << "\n";
}
```

这里最重要的不是语法，而是这个思想：

- **别手动记“什么时候解锁”，让对象析构帮你收尾**

这就是 RAII 在并发里的一个很典型落地。

---

## `unique_lock` 和“普通 `mutex`”到底差在哪

这两个东西其实不是同一层：

- `std::mutex` 是**锁本体**
- `std::unique_lock<std::mutex>` 是**管理这把锁的对象**

也就是说：

- `mutex` 负责提供 `lock()` / `unlock()`
- `unique_lock` 负责“什么时候拿锁、什么时候放锁、当前是否持锁”这套对象语义

你可以先把它理解成：

- `mutex` 像门锁
- `unique_lock` 像“拿着门锁使用权的人”

所以严格说，`unique_lock` 不是 `mutex` 的另一种版本，
而是一个**包装 `mutex` 的 RAII 锁管理器**。

---

## 为什么已经有 `mutex` 了，还要 `unique_lock`

因为只写：

```cpp
m.lock();
// ...
m.unlock();
```

太容易出问题：

- 中途 `return`
- 中途抛异常
- 分支复杂以后忘记解锁

`unique_lock` 解决的是：

- **让“锁的拥有权”变成一个对象，由对象析构自动解锁**

最基本写法：

```cpp
std::mutex m;

void work() {
    std::unique_lock<std::mutex> lock(m);
    // 临界区
}
```

离开作用域时，它会自动解锁。

---

## 它和 `lock_guard` 又差在哪

如果只看“自动加锁、自动解锁”，它们很像。

但区别是：

- `lock_guard` 很轻，很死板
- `unique_lock` 更重一点，但更灵活

`lock_guard` 基本语义就是：

- 构造时锁上
- 析构时解锁
- 中间别折腾

而 `unique_lock` 额外支持这些能力：

1. **延迟加锁**

```cpp
std::unique_lock<std::mutex> lock(m, std::defer_lock);
// 先不锁，后面再决定什么时候 lock.lock();
```

2. **手动解锁，再重新加锁**

```cpp
std::unique_lock<std::mutex> lock(m);
lock.unlock();
// 做一些不需要锁的事
lock.lock();
```

3. **尝试加锁**

```cpp
std::unique_lock<std::mutex> lock(m, std::try_to_lock);
if (!lock.owns_lock()) {
    // 这次没拿到锁
}
```

4. **可移动，不可拷贝**

这意味着“锁的拥有权”可以转移给另一个 `unique_lock` 对象。

5. **能和 `condition_variable` 配合**

这是它最重要的实战意义之一。

---

## 为什么 `condition_variable` 要求 `unique_lock`

因为条件变量在 `wait()` 时，不是傻等，而是要做一组动作：

1. 先检查条件
2. 条件不满足时，**临时把锁释放掉**
3. 当前线程睡眠
4. 被唤醒后，**重新拿回锁**
5. 再继续往下执行

这要求锁对象必须支持：

- 当前持锁状态管理
- `unlock()` 和 `lock()` 的受控切换

`lock_guard` 太死板，不提供这套接口；
`unique_lock` 正好就是为这种场景准备的。

所以你会经常看到：

```cpp
std::unique_lock<std::mutex> lock(m);
cv.wait(lock, [] { return ready; });
```

不是因为它“更高级”，
而是因为 `wait()` 的机制本来就需要这种可解锁、可再加锁的包装器。

---

## 你现在可以怎么记

先用这 3 句话把关系记死：

1. `mutex` 是锁本体，不会自己帮你管理生命周期。
2. `lock_guard` 是最简单的自动加锁/解锁包装，适合“进来就锁，出去就解”。
3. `unique_lock` 是更灵活的锁管理器，适合需要延迟加锁、临时解锁、重新加锁、配合 `condition_variable` 的场景。

如果只是普通临界区保护：

- 优先先写 `lock_guard`

如果你要控制加锁时机，或者后面要学条件变量：

- 就上 `unique_lock`

---

## `mutex` 真正保证了什么

它保证的是：

- 同一把锁保护的那段临界区，不会被多个线程同时进入

它不保证的是：

- 整个程序天然就没问题
- 不同锁之间自动有正确顺序
- 你在锁外读写同一份数据也会安全

所以判断一个并发程序对不对，不能只看“有没有 `mutex`”，而要看：

- **所有相关访问，是不是都被同一套同步规则管住了**

---

## 新手最容易犯的 4 个错

### 1. 以为“加了锁”就一定安全

如果线程 A 用的是 `m1`，线程 B 用的是 `m2`，那其实没同步到一起。

真正重要的是：

- **访问同一份共享数据的代码，要用同一把锁保护**

### 2. 锁得太小

比如你把：

```cpp
read
modify
write
```

只锁了其中一步，那还是会出问题。

你要保护的是：

- 整个不可分割的临界区

### 3. 锁得太大

如果把和共享数据无关的慢操作也锁进去，比如：

- 大量打印
- 网络 I/O
- 文件 I/O

那并发度就会很差。

### 4. 又在锁外偷偷访问同一份数据

这类 bug 很阴。

表面看你“明明加锁了”，实际上：

- 有一部分访问受保护
- 另一部分访问裸奔

那程序还是不对。

---

## 你现在该怎么理解这一章

如果把线程这一章收成一句话，就是：

- `std::thread` 负责把执行流开出来
- `mutex` 负责让共享状态别乱

前者解决的是：

- **怎么并发跑**

后者解决的是：

- **并发跑的时候怎么不把数据搞坏**

---

## 面试够用线

至少要能讲清这几句：

1. 竞态条件是结果依赖线程执行时序的并发错误
2. 多线程同时读写同一份可变数据，且没有同步，就可能产生数据竞争
3. `x++` 不是原子动作，可以拆成读、改、写
4. `mutex` 用来保护临界区，让同一时刻只有一个线程进入
5. `lock_guard` 是 RAII 风格的自动加锁/解锁包装
6. 共享同一份数据的访问，必须遵守同一套同步规则
